  - `vllm/v1/core/kv_cache_utils.py`
  - `vllm/v1/request.py`

### Summary

**What changed and why**  
Added an opt-in 'full' report mode for KV cache events that emits BlockStored events for prefix-cache-reused blocks, not just newly allocated ones. This allows external consumers (e.g., Dynamo, llm-d) to maintain an accurate global prefix tree for KV-aware routing. Changes span request parsing, block pool event emission, and KV cache manager integration.

**Technical impact**  
Introduces a new per-request parameter 'kv_cache_report_mode' (default 'incremental') that controls whether reused prefix blocks generate events. A new method emit_cached_block_events() in BlockPool emits BlockStored events without mutating block state. The KV cache manager calls this method during get_computed_blocks() when mode is 'full' and there are cache hits. Also refactors common event-building logic into _build_block_stored_event() to avoid duplication.

**Potential risks**  
Increased event volume for requests with prefix cache hits in 'full' mode, which may affect performance or downstream consumers. Preempt/reschedule may re-trigger emission for the same request; consumers must handle events idempotently. The feature is opt-in, so default behavior is unchanged. New code paths are covered by unit tests.

**Key insights**  
The PR addresses a visibility gap in the KV cache event system by reporting reused prefix blocks, enabling external systems to track the full prefix chain. The design is minimal: no block state mutation, shared event construction, and opt-in via per-request parameter. Tests cover disabled events, zero cached blocks, and correct event fields.

It enables DCP for supported attention groups while keeping non-DCP groups on local cache/state handling. Key changes include: adding `supports_dcp` flags to attention layers, modifying the flash attention backend to handle mixed decode/prefill batches with DCP, updating block table and slot mapping logic to support multiple KV cache groups with different block sizes and DCP scaling, enabling prefix caching for hybrid models, and adding tests for hybrid DCP scenarios.

**Technical impact**  
The change enables DCP for hybrid models, improving memory efficiency and throughput for large-scale inference. The FA2 split path is a workaround that should be removed once FA4 supports the required shapes. There is a risk of deadlock if DCP collectives are not taken uniformly across ranks (mitigated by rank-invariant skip logic). The changes to block table slot mapping and KV cache block size calculations could affect non-hybrid models, though tests cover basic scenarios. The prefix caching enablement for hybrid models is opt-in by default, reducing risk.

**Key insights**  
The PR introduces a clean separation between DCP-capable and non-DCP layers via the `supports_dcp` attribute on attention layers. The slot mapping mode enum (TOKEN_TO_KV_SLOT vs NONE) elegantly handles Mamba layers that don't use per-token slot mappings. The DCP context attention skip logic is carefully designed to be rank-invariant to avoid deadlocks. The test plan includes both small and large hybrid models with different DCP configurations, and accuracy results show negligible degradation (0.4% or less) with DCP enabled.
