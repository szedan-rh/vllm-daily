The vector code change is necessary because the previous vcreate approach assumed LMUL_128 to LMUL_512 directly, which fails at VLEN=256 due to fractional LMUL constraints.

## 3. [[Performance][Hardware][RISC-V] Reduce LMUL pressure in INT4 LUT dequant](https://github.com/vllm-project/vllm/pull/47538)


### Base Information

- **PR Number:** #47538
- **Author:** [I3eg1nner](https://github.com/I3eg1nner)
- **Merged By:** [bigPYJ1151](https://github.com/bigPYJ1151)
- **Merged time:** 2026-07-05 22:58:56
- **Type:** `SUBSTANTIAL`
- **[Changed Files](https://github.com/vllm-project/vllm/pull/47538/files) (1):**
  - `csrc/cpu/cpu_types_riscv_impl.hpp`

### Summary

**What changed and why**  
Refactored the INT4 LUT dequant constructor to split a 64-bit packed value into two 32-bit halves and process each with u32 vectors at LMUL_256 instead of u64 at LMUL_1024. This reduces register pressure from m8/m4 to m2/m1 on VLEN=128/256, avoiding heavy spilling and improving performance.

**Technical impact**  
Reduces per-variable register usage by 75% on both VLEN=128 and VLEN=256 configurations. On Spacemit X100 (VLEN=256), performance improves from 14.2 ns/iter to 6.8 ns/iter, a 2.08x speedup. The change is backward compatible and passes correctness tests.

**Potential risks**  
The change is specific to RISC-V vector extensions and relies on vcreate to merge halves, which may have different performance characteristics on other RVV implementations. The code assumes VLEN >= 128 and VEC_ELEM_NUM = 16; if these change, the logic may need adjustment. No functional risk as correctness is verified.

**Key insights**  
On RVV, element width directly impacts register file pressure due to LMUL grouping. Using narrower elements (u32 vs u64) for the same computation can significantly reduce spilling and improve throughput. This optimization is specific to variable-length SIMD architectures and not applicable to fixed-width SIMD like x86 AVX.

The cross-compilation change means VLEN must be manually specified via -DVLLM_RVV_VLEN when cross-compiling, which could be missed. The Python change may slightly increase latency due to the C++ call before the fast-path check, but this is negligible.

**Key insights**  
The fix addresses a chain of issues where inconsistent checks across C++, CMake, and Python led to the RVV kernel being silently disabled. The approach of using the C++ compile-time check as ground truth is sound because it reflects the actual binary capabilities. The clamping strategy is pragmatic given current kernel limitations.
