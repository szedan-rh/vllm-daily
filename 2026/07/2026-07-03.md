## 1. [[Bugfix] Fix CPU split-KV scratchpad sizing](https://github.com/vllm-project/vllm/pull/45844)


### Base Information

- **PR Number:** #45844
- **Author:** [gausah01](https://github.com/gausah01)
- **Merged By:** [bigPYJ1151](https://github.com/bigPYJ1151)
- **Merged time:** 2026-07-03 23:47:24
- **Type:** `SUBSTANTIAL`
- **[Changed Files](https://github.com/vllm-project/vllm/pull/45844/files) (2):**
  - `csrc/cpu/cpu_attn_impl.hpp`
  - `vllm/v1/attention/backends/cpu_attn.py`

### Summary

**What changed and why**  
Fixed a CPU split-KV hang by correcting scratchpad sizing. The scratchpad was previously sized using the final output element size or a smaller fallback q-head count, but the runtime split path writes temporary split outputs as float and can process more q-heads per split, causing undersizing and data overlap with reduction state. The fix sizes temporary outputs as float and uses the maximum q-head tile the runtime can use for the reduction scratchpad.

**Technical impact**  
Resolves a hang in CPU attention for models like google/gemma-4-26B-A4B-it where the worker could get stuck in reduce_splits waiting for a split completion flag. The change ensures correct scratchpad allocation, preventing memory corruption and enabling successful throughput benchmarks.

**Potential risks**  
The change increases scratchpad size calculations, which may slightly increase memory usage. However, the previous undersizing caused hangs, so this is a necessary correction. The loop over workitems and reduce_workitems adds computational overhead during scheduling, but this is negligible compared to the attention computation.

**Key insights**  
The root cause was a mismatch between scratchpad sizing assumptions and runtime behavior: the scratchpad was sized for final output element size, but split outputs are written as float and can have more q-heads per split. The fix iterates over all work items to compute the maximum required size, ensuring the scratchpad is large enough for all possible split configurations.

## 2. [[ROCm] Disable persistent sparse-MLA kernel for chunked-prefill continuations](https://github.com/vllm-project/vllm/pull/47567)


### Base Information

- **Type:** `SUBSTANTIAL`
- **[Changed Files](https://github.com/vllm-project/vllm/pull/47567/files) (1):**
  - `vllm/v1/attention/backends/mla/rocm_aiter_mla_sparse.py`

### Summary

**What changed and why**  
Added a condition to disable the persistent sparse-MLA kernel for chunked-prefill continuations on ROCm. The persistent kernel produces numerical errors for multi-token prefill batches, which compound across chunked prefill steps and cause long-context decode collapse. The fix gates the persistent path to only decode and single-chunk prefills, falling back to the correct non-persistent path for chunked continuations.

**Technical impact**  
Fixes a correctness bug where sparse-MLA models (e.g., GLM-4, DeepSeek-V3.2) on ROCm produced garbage output for contexts >20K tokens due to accumulated numerical errors in the persistent kernel during chunked prefill. Decode throughput is unaffected as the persistent path remains for decode and single-chunk prefills. The change is Python-only and only affects ROCm sparse-MLA backends.

**Potential risks**  
Low risk: the condition is conservative (disables persistent path for any chunked continuation), and the non-persistent path is known correct. There is a slight performance regression for chunked-prefill continuations, but this is necessary for correctness. The change is well-tested with needle-in-a-haystack and GSM8K evaluations.

**Key insights**  
The bug was isolated to the persistent MLA work-stealing kernel being numerically wrong for prefill-shaped batches (multi-token), while correct for decode (single-token). The error compounds through KV cache across chunked prefill steps, explaining why failure depends on chunk count rather than context length. The fix uses a simple heuristic based on query length and sequence length to decide which kernel path to use.

### Summary

**What changed and why**  
This PR enables Dynamic Speculative Decoding (DSD) with Model Runner v2 and Full Cuda Graphs (FCG). Previously DSD forced piecewise graphs; now it captures FCG for all possible decode query lengths (not just multiples of K+1) to handle cases where the optimal draft length is less than the maximum. It also decouples capture sizes for prefill and decode phases, and adds tests for DSD FCG behavior.

**Technical impact**  
The core change is in `cudagraph_utils.py` where `_init_candidates` now iterates over all decode query lengths from the DSD schedule (instead of just the max) to create FULL graph candidates. The `resolve_cudagraph_mode_and_sizes` method in `compilation.py` skips the size adjustment for MRv2 (since MRv2 handles it separately). The `_maybe_override_dynamic_sd_cudagraph_mode` in `vllm.py` no longer forces PIECEWISE when using MRv2. These changes enable FCG for DSD on MRv2, improving performance (TPOT from 2.97ms to 2.91ms in benchmarks).

**Potential risks**  
Capturing more graph shapes increases memory usage and compilation time. The change is gated behind MRv2 and DSD, so non-DSD or MRv1 users are unaffected. There is a risk of duplicate graph entries (handled by dedup check). The new logic relies on `num_speculative_tokens_per_batch_size` being set, which is guaranteed by DSD but could cause errors if misconfigured. The tests cover uniform and non-uniform decode paths, but edge cases with large batch sizes or many query lengths may expose performance regressions.

**Key insights**  
The PR decouples capture sizes for prefill and decode, addressing a previous limitation where decode graph size dictated prefill capture. The approach of capturing all possible decode query lengths for DSD is a pragmatic solution to avoid missing graphs when the optimal draft length varies. The benchmark shows FCG matches the performance of basic SD (2.91ms) while DSD with PW was slower (2.97ms), confirming the benefit.

  - `vllm/v1/worker/gpu/attn_utils.py`
  - `vllm/v1/worker/gpu_model_runner.py`

### Summary

**What changed and why**  
Added support for NVFP4 KV cache with --kv-cache-dtype-skip-layers sliding_window. Previously, NVFP4 quantization did not work when skip layers (e.g., sliding window) were specified. The change introduces a new alignment phase in the platform interface to handle heterogeneous KV dtypes sharing a block pool, updates the sliding window spec to pick an optimal block size, and adjusts KV cache shape computation to use the correct dtype per layer (quantized vs unquantized). Also updated documentation and added a new config field skip_page_size_padded.

**Technical impact**  
Enables NVFP4 KV cache quantization with skip layers, which was previously unsupported. The block size may be automatically increased to accommodate larger per-token pages of unquantized skip layers, potentially increasing memory usage but ensuring correctness. The change modifies the block size alignment logic and KV cache shape computation to handle mixed dtypes per layer.

**Potential risks**  
Increased memory usage due to larger block sizes when skip layers are present. The new alignment logic may interact with existing hybrid model block size alignment; careful testing is needed. The change assumes that the primary quantized dtype has a smaller per-token page than the unquantized skip layers; if not, an assertion is triggered. The new config field skip_page_size_padded is not part of the hash for prefix caching, which could cause subtle issues if the config changes.

**Key insights**  
The solution uses a two-phase approach: first align block sizes so that the quantized primary's page covers the largest padded-spec page, then record the shared page size in a new config field so that unify_kv_cache_spec_page_size can reconcile specs trivially. The sliding window spec now dynamically selects the largest kernel block that fits the shared page to minimize padding waste. The per-layer dtype selection in KV cache shape computation ensures that skipped layers use the unquantized layout.

The change is low-risk because it only adds a condition to an existing socket option. The PR correctly limits the condition to multi-worker serving, which is the intended use case.

**Key insights**  
The PR addresses a subtle footgun where SO_REUSEPORT could cause silent port sharing between unrelated vLLM instances. By making the option conditional, it aligns with the principle of least privilege. The change is minimal and well-targeted, affecting only the socket creation logic.

Refactored GLM4V model to avoid initializing the full HuggingFace processor during startup metadata reads. Reduces startup latency and memory usage for GLM4V models by avoiding full processor initialization in metadata-only paths. **Potential risks**  
Risk of mismatch between config-based processor class name and actual processor class if config is outdated or incorrect. The fallback to get_hf_processor() when processor_class_name is None mitigates this. Also, the new methods rely on config files being present and correctly formatted; missing or malformed configs could raise errors. The change assumes that processor config files are accessible and contain 'size' with 'longest_edge'.

**Key insights**  
Startup optimization by deferring heavy processor initialization to only when needed. The approach of reading metadata from config files instead of instantiating full processors can be applied to other models. The performance gains are significant (seconds and hundreds of MiB) and validated with benchmarks. The change maintains backward compatibility by keeping the original path as fallback.
