### Summary

**What changed and why**

**Key insights**  
The fix addresses a security gap where derender endpoints bypassed resource limits enforced in the normal generation path. Validation is placed early in the request handling pipeline to minimize wasted work. The negative token_ids validator leverages Pydantic's built-in validation, ensuring consistency across all request parsing.

### Summary

**What changed and why**  
Added a new function `_align_mixed_attention_kv_cache_views` and a helper `_restride_blocks_first_kv_cache_to_kv_first_storage` to fix a regression in encoder-decoder models on ROCm. The V2 runner's `_reshape_kv_cache` previously left shared KV cache allocations with mixed layout semantics (K/V-first for decoder self-attention vs blocks-first for cross-attention), causing block IDs to address different physical bytes and produce garbage output. The fix detects when backends disagree on block dimension and re-strides the blocks-first views to match the K/V-first storage layout.

**Technical impact**  
Ensures correct KV cache sharing between decoder self-attention and cross-attention in encoder-decoder models on ROCm with the V2 runner. The change is localized to `attn_utils.py` and only activates when mixed block dimensions are detected, so it has no impact on models without such sharing. The re-striding operation modifies the tensor's stride in-place, which is efficient and avoids data movement.

**Potential risks**  
The re-striding assumes the tensor is contiguous in the tail dimensions (NHD layout); if a non-NHD layout is encountered, it logs a warning and skips alignment, which could leave the bug unfixed for those cases. The change only applies when `kv_cache_config` is not None and `has_attn` is true, so it won't affect non-attention layers. There is a small risk of incorrect stride computation if the tensor shape assumptions are violated, but assertions and the warning mitigate this.

**Key insights**  
The root cause was a mismatch in physical layout interpretation between ROCm's attention backend (K/V-first) and other backends (blocks-first) when sharing a single KV cache allocation. The fix elegantly re-strides the blocks-first views to match the K/V-first storage without copying data. This highlights the importance of aligning tensor layouts when sharing memory across different attention backends.

  - `vllm/model_executor/models/diffusion_gemma.py`

### Summary

**What changed and why**  
The DiffusionGemma sampler's transient logit memory is bounded by tiling the compiled sample step over request groups. The group size is auto-sized from free GPU memory to prevent OOM at high concurrency. The change moves clearing of sampled/num_sampled outside the loop and computes logprobs per tile to free intermediate buffers. This reduces transient memory from O(NumDecode * Canvas * Vocab) to O(group * Canvas * Vocab).

**Technical impact**  
Reduces sampler transient memory from 23.9 GiB to 5.5 GiB, allowing max_num_seqs=32 where previously OOM occurred. Throughput and GSM8K accuracy remain at parity (54.8% vs 54.5%). The tiling is bit-identical to a single pass, ensuring no numerical changes. The change introduces a dependency on current_platform.mem_get_info() for dynamic group sizing.

**Potential risks**  
The dynamic group sizing relies on free GPU memory, which may vary during execution and could lead to suboptimal tiling or fragmentation. The budget calculation uses a heuristic (10x factor) that may need tuning for different hardware or workloads. The change moves clearing of sampled/num_sampled outside the loop, which could cause issues if the loop is not properly scoped. The logprobs computation per tile may introduce overhead for small batch sizes.

**Key insights**  
The tiling approach effectively bounds transient memory without sacrificing accuracy or throughput. The use of free memory for auto-sizing avoids manual tuning knobs. The bit-identical property ensures correctness. The change is a targeted optimization for high-concurrency scenarios where the sampler was the memory bottleneck.
