### Summary

**What changed and why**  
Changed `resolve_items` in `AsyncMultiModalItemTracker` to use `return_exceptions=True` in `asyncio.gather` and then raise the first exception after all tasks complete, preventing orphaned in-flight tasks on partial failure. Previously, a single failing fetch would cause `gather` to propagate the exception immediately, leaving sibling fetches running detached and holding resources.

**Technical impact**  
Fixes a resource leak where network sockets and thread pool workers could be orphaned when one media fetch in a batch fails. The fix ensures all tasks in a modality are awaited before raising, at the cost of waiting for the slowest sibling fetch before surfacing an error. This is the correct tradeoff to avoid abandoning in-flight work.

**Potential risks**  
Low risk. The change only affects error handling in `resolve_items`; normal success paths are unchanged. The only behavioral difference is that on partial failure, the error is raised after all tasks complete instead of immediately, which may slightly increase latency in error cases but prevents resource leaks.

**Key insights**  
The bug was a classic `asyncio.gather` pitfall: using `return_exceptions=False` (default) causes immediate exception propagation, leaving other tasks orphaned. The fix uses `return_exceptions=True` and manual exception checking to ensure all tasks complete. The test uses `asyncio.all_tasks()` diffing to verify no tasks leak, which is a robust way to detect orphaned tasks.
