  - `tests/entrypoints/serve/instrumentator/test_http_status_metrics.py`
  - `vllm/entrypoints/openai/api_server.py`

### Summary

**What changed and why**  
Fixed a bug where Prometheus http_requests_total metric incorrectly recorded certain 4xx client errors (ValueError, TypeError, OverflowError, VLLMValidationError, VLLMNotFoundError, NotImplementedError) as 5xx. The root cause was that Starlette's middleware architecture places exception handlers for specific types in ExceptionMiddleware (inner layer) and handlers for Exception in ServerErrorMiddleware (outer layer). The Prometheus middleware sits between these layers, so unhandled specific exceptions propagated through it and defaulted to status 500. The fix registers these specific exception types as individual handlers so they are caught by ExceptionMiddleware before reaching Prometheus, ensuring correct status code recording.

**Technical impact**  
Improves accuracy of the http_requests_total metric by ensuring that client errors (4xx) are correctly categorized instead of being misreported as server errors (5xx). This reduces noise in monitoring and alerting systems, allowing operators to distinguish between client and server issues. The change also adds a comprehensive test suite (10 test cases) covering various exception types and status groups, ensuring regression prevention.

**Potential risks**  
Low risk. The change only adds exception handlers for specific types that were previously unhandled; the behavior for other exceptions remains unchanged. The new handlers delegate to the existing exception_handler function, so no new error handling logic is introduced. The test suite validates both correct and incorrect status groups, reducing the chance of regressions. However, if any of these exception types are intentionally expected to result in a 5xx status in some edge case, the fix would change that behavior. Based on the PR description, all listed exceptions should result in 4xx responses, so this is unlikely.

**Key insights**  
The bug was caused by a subtle interaction between Starlette's middleware layers and the Prometheus instrumentation library. The fix demonstrates the importance of understanding middleware ordering when integrating monitoring tools. The test suite is well-structured, using parametrized tests to cover multiple exception types and verifying both the HTTP response code and the Prometheus metric. The PR also highlights that VLLMValidationError was already fixed in a separate PR, but the other exceptions were missed until now.

### Summary

**What changed and why**  
Two performance optimizations for MXFP8 kernels on AMD MI355X (gfx950): (1) a shape-adaptive tile selector replaces the fixed two-bucket launcher in the dense-linear GEMM, improving occupancy and bandwidth efficiency across varying M, N, K shapes including TP-sharded layouts; (2) a GROUP_SIZE_M super-grouping swizzle in the grouped-MoE GEMM reduces redundant A re-reads by keeping expert data resident across N-tiles. **Technical impact**

The K-divisibility guards ensure correctness but may silently fall back to suboptimal tiles for unsupported K values. The MoE kernel now uses a 1D grid with swizzle, which could interact poorly with future Triton changes. The test is gated on gfx950, so regressions on other ROCm GPUs may go undetected.

**Key insights**  
The shape-adaptive tile selector demonstrates that fixed tile sizes are suboptimal for varying M, N, K, especially with TP sharding. The MoE swizzle effectively reduces memory traffic by exploiting data reuse across N-tiles, a technique applicable to other grouped GEMM scenarios. The performance gains are achieved without changing the underlying math or adding new quantization kernels, relying solely on launch configuration and grid layout changes.

  - `vllm/models/minimax_m3/nvidia/sparse_attention_msa.py`

### Summary

**What changed and why**  
Replaced two separate Triton top-k kernels (one for decode, one for prefill) in the MiniMax-M3 MSA indexer with a single MSA `sparse_topk_select` over a unified score buffer. This reduces kernel launches, simplifies buffer management, and leverages a single optimized MSA path for both prefill and decode.

**Technical impact**  
The change introduces a unified score buffer `[total_q, H, max_k_tiles]` shared by decode and prefill. Decode scores are computed via a new `minimax_m3_index_decode_score` kernel (split-K, cudagraph-safe), prefill scores via fmha `OnlyScore` writing directly into the buffer, and a single `sparse_topk_select` selects top-k blocks. The `topk_indices_buffer` layout changed from head-major to token-major. Sentinel-based valid block counting in sparse attention kernels was replaced with causal position-based counting. The MSA pin was bumped to include the `sparse_topk_select` entry point.

**Potential risks**  
The unified buffer has a hardcoded `MAX_K_TILES` of 8192, which may be insufficient for very long sequences (>1M tokens) and could cause out-of-bounds writes if exceeded. The removal of sentinel padding in sparse attention kernels assumes valid block count is always derivable from sequence position, which may break if top-k indices are not contiguous or if padding is needed for other reasons. The change requires cudagraph compatibility and stable buffer addresses, which could be fragile if buffer shapes change. The test environment lacks HF authentication for some tests, so full regression coverage is not confirmed.

**Key insights**  
Consolidating two top-k paths into one reduces code duplication and kernel launch overhead. The token-major buffer layout aligns with MSA's native output format, simplifying data flow. The use of a persistent buffer with sentinel fill ensures cudagraph replay safety. The change demonstrates a pattern of leveraging MSA's `sparse_topk_select` for unified top-k selection across different score computation backends.
