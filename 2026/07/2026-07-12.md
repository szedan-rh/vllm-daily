## 1. [[ROCm][CI] Cache Rust builds by source inputs](https://github.com/vllm-project/vllm/pull/46527)


### Base Information

- **PR Number:** #46527
- **Author:** [AndreasKaratzas](https://github.com/AndreasKaratzas)
- **Merged By:** [AndreasKaratzas](https://github.com/AndreasKaratzas)

- **Type:** `SUBSTANTIAL`
- **[Changed Files](https://github.com/vllm-project/vllm/pull/46527/files) (3):**
  - `.buildkite/scripts/ci-bake-rocm.sh`
  - `docker/Dockerfile.rocm`
  - `docker/ci-rocm.hcl`

### Summary

**What changed and why**  
Added a content-addressed Rust build cache for ROCm CI to avoid recompiling Rust when only non-Rust files change. The Rust build is split into narrow input stages (rust-toolchain-input, rust-input) that copy only Rust-related sources, toolchain files, and build scripts. A dedicated rust-rocm-ci cache-only target is added, and the ci_base image now includes the Rust/protoc toolchain. The wrapper script computes a content hash from the Rust inputs and uses it to import/export cache refs, so non-Rust changes reuse cached cargo output.

**Technical impact**  
Reduces CI build time by ~5 minutes on average when Rust sources are unchanged, because the cargo build layer is reused from a content-addressed cache. The Rust toolchain is baked into ci_base, so per-commit images no longer need to fetch and install Rust/protoc. The change adds new Dockerfile stages and cache functions, increasing complexity but improving cache granularity. The ci_base content hash now includes Rust toolchain files, which may cause ci_base rebuilds when those files change.

**Potential risks**  
The sparse-checkout and COPY logic must correctly include all Rust source files; missing files could cause build failures. The content hash computation must be deterministic and cover all inputs; any mismatch could lead to cache misses or stale builds. The new cache export mode (ROCM_CONTENT_CACHE_EXPORT_MODE) adds configuration complexity. If the Rust cache is incorrectly invalidated, CI may rebuild Rust unnecessarily. The ci_base image size increases due to the Rust toolchain (~1-2 GB).

**Key insights**  
The change uses a content-addressed cache key based on Rust-specific inputs, which is a standard pattern for incremental builds. Splitting the toolchain and source inputs into separate stages allows the toolchain to be cached in ci_base while source changes only invalidate the rust-build stage. The sparse-checkout approach for remote builds avoids cloning the entire repo, reducing network and disk usage. The cache export mode (missing/always/never) provides flexibility for debugging and CI tuning.

## 3. [[CPU] Create Proper Numa topology for s390x](https://github.com/vllm-project/vllm/pull/40714)

  - `vllm/utils/cpu_resource_utils.py`
  - `vllm/utils/ompmultiprocessing.py`
  - `vllm/v1/worker/cpu_worker.py`

### Summary

**What changed and why**  
This PR introduces proper NUMA topology support for s390x architecture. Previously, only sockets were considered for CPU grouping, but s390x has a hierarchy of drawer > book > socket > core. The changes modify the CPU resource utilities to detect s390x and use book or socket as the grouping key for NUMA nodes, improving performance in throughput-oriented workloads. Additionally, the Dockerfile for s390x is heavily refactored: it removes the custom Apache Arrow build, updates Numba/llvmlite versions, removes the guidance-builder stage, and simplifies the build process. The llguidance dependency is also enabled for s390x.

**Technical impact**  
The CPU topology detection logic in `vllm/utils/cpu_resource_utils.py` is extended to handle s390x by parsing book and socket information from lscpu and selecting the best grouping level. This changes how NUMA nodes are assigned to CPUs, affecting thread binding and memory allocation. The Dockerfile changes significantly reduce build complexity and dependencies, potentially speeding up image creation and reducing maintenance. The llguidance library is now allowed on s390x, enabling structured output features. The CPU worker and communicator are updated to handle cases where the synthetic NUMA node (book ID) does not correspond to a real memory node, falling back to the first available memory node.

**Potential risks**  
The s390x-specific topology logic may not handle all hardware configurations correctly, especially if book or socket values are missing or inconsistent. The fallback to the first memory node could lead to suboptimal memory locality if the synthetic node does not match the actual memory hierarchy. The Dockerfile changes remove the custom Apache Arrow build, which may introduce compatibility issues if the pre-built Arrow wheel is not available or has different features. The removal of the guidance-builder stage assumes the upstream llguidance fix is merged, which may not be the case. The update to Numba 0.65.0 and llvmlite 0.47.0 could introduce regressions or incompatibilities with other dependencies.

**Key insights**  
The PR addresses a real performance issue on s390x by aligning CPU topology detection with the actual hardware hierarchy. The approach of using book/socket as a synthetic NUMA node is pragmatic but requires careful fallback handling. The Dockerfile simplification is a significant cleanup that reduces build time and complexity, but it relies on external wheels being available. The changes are well-documented with comments explaining the s390x-specific logic. The test results show successful book-aware binding, indicating the topology detection works as intended.

### Summary

**What changed and why**  
Added support for DFlash speculative decoding on the CPU backend for Gated DeltaNet (GDN) hybrid models. This includes a new CPU kernel for fused sigmoid gating delta rule update with speculative decoding, modifications to the GDN attention layer to handle wide conv-state buffers and multi-slot SSM state, and stashing of unpacked conv weights for the torch-based spec path. The change was necessary because the existing CPU implementation did not support the speculative decoding shapes and hooks required by DFlash.

**Technical impact**  
Enables DFlash speculative decoding on CPU for GDN models, improving throughput and latency. The new kernel processes a varlen batch of draft tokens sequentially per sequence, reading initial state from a cache slot and storing updated state per token for rollback. The attention layer now distinguishes between spec and non-spec tokens, using torch conv for the wide rolling buffer and the new kernel for SSM updates. The change also adds a Python binding and a custom op registration.

**Potential risks**  
The new code path is only exercised when speculative decoding is configured; otherwise the original fast path is used. However, the wide conv buffer allocation and multi-slot SSM state may increase memory usage. The torch-based conv path for spec tokens may be slower than the AMX kernel, but this is acceptable for the spec decode scenario. The changes are limited to CPU backend and GDN models, so no risk to other backends or model types.

**Key insights**  
The implementation mirrors the GPU spec decode path but uses CPU-specific kernels and torch operations. The conv-state cache is widened to accommodate draft tokens, and the SSM state is stored per draft token position to support rollback. The unpacked conv weight is stashed to avoid VNNI packing issues when using torch conv. The PR includes detailed test results showing successful speculative decoding with acceptance rates and throughput metrics.

### Summary

**What changed and why**  
Renamed `instance_id` parameter to `engine_id` in `SharedOffloadRegion` and its tests to reflect that the identifier is now per-engine (DP-replica) rather than per-instance. In `spec.py`, the scheduler and worker now use `engine_id` from `kv_transfer_config` instead of `vllm_config.instance_id`, and the worker rank is computed as `current_device_index() % world_size` to map global device indices to replica-local slots. **Technical impact**

If `engine_id` is not set or is not unique per replica, offload regions could collide. The assertion in `__init__` guards against `None`, but uniqueness must be ensured by the caller. The rank modulo operation assumes that device indices are contiguous per replica; if not, it could cause incorrect slot mapping. Existing tests were updated to use `engine_id` instead of `instance_id`, but no new tests were added for DP scenarios.

**Key insights**  
The change makes the offload tier DP-aware by using a per-replica engine ID and local rank mapping. This is a necessary fix for multi-replica setups where each replica has its own KV cache and must not share offload regions. The rename from `instance_id` to `engine_id` clarifies the semantics.

**Potential risks**  
Low risk. The change adds a guard that was already present in similar non-residual and residual patterns. The regression test covers the mixed-dtype case. Performance impact is positive or neutral as shown by benchmarks.

**Key insights**  
The bug was introduced in #21069 when adding residual quantized patterns without the dtype check that existed in other patterns. The fix is minimal and targeted, and the test ensures the guard works correctly for Gemma-style RMSNorm with mixed dtypes.
