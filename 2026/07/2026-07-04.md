**Key insights**  
The bug was caused by a missing forwarding of the `instruction` field from `chat_template_kwargs` to the prompt formatter. The fix demonstrates the importance of ensuring all request parameters are propagated through the processing pipeline. The test plan includes a clear before/after comparison showing the fix works as expected.

### Summary

**What changed and why**  
Fixed two bugs in `_make_mm_prefix_mask_mod` for FA4 backend: (1) missing sliding window constraint on the causal term, causing incorrect attention for long sequences; (2) using local `q_idx` instead of absolute position, causing wrong masks in chunked prefill. Added `sliding_window_left` parameter and absolute position computation via `seqlen_info`.

**Technical impact**  
Corrects attention masking for Gemma4 vision models on FA4 (SM90+), enabling correct output for long contexts with images. Affects only models with `is_mm_prefix_lm=True` on FA4; other paths unchanged. Introduces two `@cute.jit` branches to avoid runtime branching.

**Potential risks**  
Low risk: changes are scoped to a single function with clear guard conditions. The `q_abs` fix may affect other mm-prefix models, but they should benefit from correct absolute positions. The sliding window branch is only taken when `sliding_window_left` is not None, which is only set for Gemma4 local layers.

**Key insights**  
The root cause was subtle: FA4's `mask_mod` replaces kernel-level window enforcement, so the mask must explicitly include the sliding window term. Also, `q_idx` is chunk-local while `kv_idx` is absolute, requiring manual offset correction. The fix aligns with Triton's reference implementation.

Adds 480 lines of code across 6 files, including a new test file with 16 unit tests. Modifies grammar_bitmask to detect reasoning-end mid-window and constrain subsequent positions, fixes bonus row bitmask to use OR of should_fill_bitmask and apply_bitmask, tolerates accept_tokens rejection for post-marker drafts, and introduces reasoning_end_token_index tracking with trim_reasoning_for_advance to strip reasoning content before grammar advancement in the scheduler. All changes are backward-compatible and pass existing tests.

**Potential risks**  
The trim_reasoning_for_advance logic relies on correct reasoning_end_token_index, which could be off if the reasoning-end marker is multi-token or detected late; fallback to last index is conservative but may drop valid grammar tokens. The mid-window detection uses is_reasoning_end_streaming which may have performance overhead for large windows. The rollback of grammar state after mid-window advancement could interact poorly with other stateful operations. The changes are specific to speculative decoding with structured output and should not affect other paths.

**Key insights**  
The root cause is that speculative decoding processes multiple tokens per step, and the reasoning-end marker can appear mid-window, causing bitmask and grammar advancement to be inconsistent. The fix ensures that once the marker is detected, all subsequent positions in the same step are constrained by the grammar, and reasoning content is never fed to the grammar. The bonus row fix prevents -1 padding from accidentally disabling bitmask. The hardening tolerates invalid drafts that predate grammar activation. The trim_reasoning_for_advance complements #42452 by ensuring the scheduler only advances grammar with non-reasoning tokens.
