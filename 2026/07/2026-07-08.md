## 1. [[KV Offloading] Add free block iterator for CPU offload scheduling](https://github.com/vllm-project/vllm/pull/47849)


### Base Information

- **PR Number:** #47849
- **Author:** [chaunceyjiang](https://github.com/chaunceyjiang)

- **Type:** `SUBSTANTIAL`
- **[Changed Files](https://github.com/vllm-project/vllm/pull/47849/files) (2):**
  - `vllm/v1/core/kv_cache_utils.py`
  - `vllm/v1/simple_kv_offload/manager.py`

### Summary

**What changed and why**  
Added a new method `iter_blocks_after` to `FreeKVCacheBlockQueue` that yields free blocks in eviction order starting after a given cursor. Updated `SimpleCPUOffloadScheduler._prepare_lazy_store_specs` to use this iterator instead of directly accessing internal linked list pointers (`fake_free_list_head`, `fake_free_list_tail`, `next_free_block`). This decouples the scheduler from the queue's implementation details, enabling support for alternative queue implementations like `PriorityAwareFreeKVCacheBlockQueue`.

**Technical impact**  
Reduces coupling between the CPU offload scheduler and the free block queue's internal structure. The scheduler now relies on a stable interface (`iter_blocks_after`) rather than traversing the linked list manually. This makes the code more maintainable and extensible for future queue variants. The change is backward-compatible as the new method encapsulates the same traversal logic.

**Potential risks**  
Low risk. The new iterator replicates the existing traversal logic exactly. However, if any other code path relies on the scheduler's direct access to queue internals (none in this PR), it would break. The `covered` variable was previously used as a counter but is now derived from `enumerate`; this is functionally identical. Edge cases (e.g., cursor at tail, empty list) are handled by the iterator's while condition.

**Key insights**  
The PR addresses a design smell where the scheduler was tightly coupled to the queue's linked list implementation. Introducing an iterator is a clean refactoring pattern that improves modularity. The change is minimal (+18/-17 lines) but has significant architectural benefit for upcoming queue implementations.

## 2. [[XPU][LoRA] Fix torch.compile DEVICE_LOST by avoiding view-mutation in LoRA shrink](https://github.com/vllm-project/vllm/pull/47944)


### Base Information

- **PR Number:** #47944
- **Author:** [chaojun-zhang](https://github.com/chaojun-zhang)

- **[Changed Files](https://github.com/vllm-project/vllm/pull/47944/files) (1):**
  - `vllm/lora/punica_wrapper/punica_xpu.py`

### Summary

**What changed and why**  
Fixed DEVICE_LOST error in torch.compile by avoiding view-mutation in LoRA shrink. In `_apply_shrink`, replaced direct mutation of output tensor views via `bgmv_shrink` with a temporary buffer `buf` and then `y.copy_(buf)`. Also added early return in `add_lora_fused_moe` when `no_lora_flag` is true, aligning with GPU wrapper behavior.

**Technical impact**  
Prevents SYCL kernels from accessing freed memory due to `auto_functionalize` failing to track view lifetimes, thus eliminating `UR_RESULT_ERROR_DEVICE_LOST` crashes. The temporary buffer ensures safe write-back without view aliasing. The early return avoids unnecessary computation when no LoRA is needed.

**Potential risks**  
The temporary buffer allocation adds a small memory overhead and a copy operation, which may slightly increase memory usage and latency. However, the buffer is small (size of output slice) and the copy is negligible compared to the kernel execution. The early return change is safe as it matches the GPU wrapper logic.

**Key insights**  
The root cause was that `auto_functionalize` cannot handle view mutations in custom ops, leading to use-after-free on XPU. Using a temporary buffer decouples the custom op from the output tensor's memory management. The fix is minimal and targeted, addressing the specific failure mode without broader changes.

  - `vllm/model_executor/layers/fused_moe/routed_experts.py`

### Summary

**What changed and why**  
Fixed a double-transpose bug in fused MoE expert weight loading where a single fused tensor (gate_up_proj) was matched twice for w1 and w3, and the transpose in the w1 iteration mutated the shared loaded_weight variable, causing w3 to see an already-transposed tensor and transpose it again. This led to incorrect chunking and a crash for block-scale tensors. The fix introduces a local variable fused_weight to avoid mutating loaded_weight across iterations.

**Technical impact**  
Resolves a crash when loading fused MoE expert weights with transposed checkpoint layout (e.g., Qwen3-VL-30B-A3B-Instruct-FP8). The fix ensures both w1 and w3 start from the original tensor, making the transpose operations symmetric and correct. The w2 and non-fused paths are unchanged.

**Potential risks**  
Low risk. The change is localized to the fused weight loading path and only affects the handling of shared tensors. The logic for w2 and non-fused paths remains identical. The fix has been tested on ROCm with the target model and passes end-to-end tests.

**Key insights**  
The bug was caused by mutating a shared variable (loaded_weight) across loop iterations. Using a local copy per iteration prevents unintended side effects. The plain weight survived by luck because after the first transpose its last dimension matched hidden, so the second transpose was skipped, but block-scale tensors did not have this property.
