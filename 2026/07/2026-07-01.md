  - `vllm/model_executor/layers/fused_moe/experts/flashinfer_cutlass_moe.py`

### Summary

**What changed and why**  
Removed the assignment of `self.max_capture_size` from `moe_config.max_capture_size` and stopped passing `tune_max_num_tokens` to the FlashInfer CUTLASS MoE kernel. Previously, the CUDA graph max capture size was used as the tuning token bound, which could be unrelated to the actual MoE token limit and cause suboptimal performance. This change reverts to the default `tune_max_num_tokens` value (8192) from FlashInfer, restoring the behavior before PR #34542.

**Technical impact**  
The MoE kernel tuning will now use the default token bound (8192) instead of the CUDA graph capture size. This can improve performance for models where the capture size was too small or mismatched, as shown by throughput gains of up to 12% on Mixtral-8x7B and DeepSeek-V2-Chat. The change is minimal (2 lines removed) and only affects the FlashInfer CUTLASS MoE backend.

**Potential risks**  
Low risk. The change reverts to a previously working default value. However, if a model's actual max token count exceeds 8192, the tuning may not be optimal for larger batches. The PR acknowledges that the ideal value depends on parallel factor, which is hard to determine, so the default is a safe compromise.

**Key insights**  
Using CUDA graph capture size as a tuning parameter for MoE kernels was incorrect because it describes graph capacity, not token limits. This fix highlights the importance of using semantically correct parameters for kernel tuning. The performance improvements demonstrate that the previous change caused regressions, and reverting to defaults is a pragmatic solution until a more precise value can be derived.

  - `vllm/_aiter_ops.py`
  - `vllm/compilation/passes/fusion/rocm_aiter_fusion.py`

### Summary

**What changed and why**  
Added a new fusion pattern (MLADualRMSPerTokenQuantPattern) to fuse the asymmetric pair of q-latent RMSNorm+FP8 per-token quant and kv-latent RMSNorm into a single AITER kernel (fused_qk_rmsnorm_per_token_quant). This restores the dual RMSNorm fusion for FP8 attention paths where the earlier RMSNorm+Quant fusion had broken the symmetric pattern. Also added the corresponding custom op, fake op, availability guard, documentation, and tests.

**Technical impact**  
Introduces a new custom op (fused_mla_dual_rms_norm_per_token_quant) and a new pattern matcher that reduces two kernel launches (q norm+quant and kv norm) into one, improving throughput by ~1-2% and reducing TPOT by ~1-2% for FP8 MLA attention on ROCm. The change is guarded by AITER kernel availability and only activates when fuse_mla_dual_rms_norm is enabled (O1+).

**Potential risks**  
The new pattern relies on specific AITER kernel availability; if the kernel is missing, the pattern is a no-op. The pattern matching is sensitive to the exact graph structure (split order, dynamic quant op), so changes in upstream passes could break the match. The test uses a high tolerance (1e-2) for kv_normed and checks >99% bit-exact for q, which may mask minor numerical differences. The FP8 per-token quant scale shape is (M,1) which must match the downstream GEMM expectation.

**Key insights**  
The PR addresses a silent fusion regression where an earlier pass (RocmAiterRMSNormQuantFusionPass) folds q norm+quant into a single op, breaking the symmetric dual norm pattern. The solution is to match the asymmetric pair and lower it to a new fused op. The design is clean: it reuses the existing pass config and adds a guard for kernel availability. The performance gains are modest but consistent across various concurrency levels and sequence lengths.

**Technical impact**  
Improves acceptance rates for positions > 0 in MTP speculative decoding, increasing overall throughput and reducing latency. The fix ensures correct topk indices are used for all draft steps, leading to higher quality draft tokens and better acceptance rates.

**Potential risks**  
Low risk. The change is localized to the MTP model and proposer, and only affects the path where topk indices are shared. The compaction operation is a simple gather that does not change the buffer size or introduce new dependencies.

**Key insights**  
The bug was caused by a mismatch in the layout of topk indices between the first draft step (which writes indices for all tokens) and subsequent steps (which expect only the last token per request). The fix compacts the buffer to the correct layout, which is a minimal and efficient solution.

### Summary

**What changed and why**  
Fixed two Triton attention kernels to avoid loading KV cache slots beyond seq_len, which could contain NaN/garbage from unwritten block tails. The over-read was masked by score masking but 0*NaN still poisons the output. The fix clamps tile loop bounds to seq_len and adds a load mask to exclude out-of-range slots.

**Technical impact**  
Prevents NaN propagation in paged attention kernels when the last partial block's unwritten tail contains non-finite values. This bug surfaced as Whisper cross-attention garbage on ROCm under MRV2. The fix ensures correctness regardless of memory initialization, as reading beyond seq_len is never valid.

**Potential risks**  
Low risk: the changes are minimal and only affect edge cases where seq_len is not a multiple of block size. The added mask may slightly increase instruction count but no performance regression is expected. The fix is conservative and matches the existing score mask behavior.

**Key insights**  
The root cause was that Triton kernels relied on score masking to ignore out-of-range KV cache slots, but 0*NaN = NaN breaks that assumption. The fix aligns load masks with the score mask, ensuring no out-of-range data is loaded. The bug was triggered by MRV2's dummy warmup producing NaN in unwritten block tails.

### Summary

**What changed and why**  
Fixed a bug in beam search candidate indexing when logprobs count varies across beams. Previously, the code assumed all beams produce the same number of logprobs, leading to an IndexError when a beam had fewer logprobs. The fix restructures candidate collection into a list of tuples and uses numpy operations on that list instead of relying on a fixed logprobs_num divisor.

**Technical impact**  
Beam search now correctly handles cases where different beams produce different numbers of logprob candidates (e.g., due to varying vocabulary sizes or sampling). The change eliminates the logprobs_num variable and the flawed integer division indexing, replacing it with direct candidate tracking. A new unit test validates the fix.

**Potential risks**  
The refactored logic changes the selection of top-k candidates from numpy argpartition on a flat array to argpartition on a dynamically sized candidate list. Performance may slightly differ, but correctness is improved. The new code also breaks out of the loop if no candidates remain, which could change behavior in edge cases where all beams produce EOS tokens.

**Key insights**  
The root cause was using integer division (idx // logprobs_num) to map candidate indices back to beams, which fails when beams have different numbers of logprobs. The fix aggregates candidates with their source beam and logprobs, then processes them uniformly. The test uses a mock engine with extra logprob candidates to ensure the fix works.
