## 1. [[ROCm][perf] Use workspace manager for sparse indexer allocations](https://github.com/vllm-project/vllm/pull/41002)


### Base Information

- **PR Number:** #41002
- **Author:** [tuukkjs](https://github.com/tuukkjs)
- **Merged By:** [vllm-bot](https://github.com/vllm-bot)

- **Type:** `SUBSTANTIAL`
- **[Changed Files](https://github.com/vllm-project/vllm/pull/41002/files) (1):**
  - `vllm/v1/attention/ops/rocm_aiter_mla_sparse.py`

### Summary

**What changed and why**  
Replaced dynamic per-call torch.empty/torch.full allocations in the ROCm sparse attention indexer with workspace manager allocations for prefill k_fp8/k_scale buffers and decode logits buffers. This reduces allocator churn and matches the CUDA sparse indexer pattern. Added memory profiling early-exit path to reserve workspace sizes upfront.

**Technical impact**  
Reduces repeated PyTorch allocator calls in hot paths, lowering risk of fragmentation and allocation failures. Makes ROCm behavior consistent with CUDA. Profiling reserves scratch memory once, avoiding late allocations. Changes affect only ROCm AITER MLA sparse attention backend.

**Potential risks**  
Workspace manager may not be available in all contexts (e.g., during torch.compile FakeTensor mode), but the code guards against this by only reserving in the real implementation. If workspace manager fails to allocate sufficient memory, it could cause runtime errors. The change is limited to ROCm and should not affect other backends.

**Key insights**  
The PR is a robustness and consistency improvement, not a bug fix for the previously observed MTP crashes. It leverages existing workspace manager infrastructure to manage scratch buffers, reducing allocator overhead. The test results show no regressions and successful runs across MTP configurations.

Regression is unlikely as the default quant_config=None is already used for non-quantized models. However, if any downstream code relies on the vision embedder being quantized (which was never intended), it may break.

**Key insights**  
The bug was caused by a mismatch between the order of operations: quant_config was passed to ColumnParallelLinear before the ignore list was applied via apply_vllm_mapper. Passing the correct prefix ensures the ignore check works properly. The fix aligns the unified variant with the tower-based variant, which never quantizes the vision tower.

The main changes are: (1) prepending non-cached blocks to the free queue while appending cached blocks, so cached blocks are deprioritized for reuse and survive longer; (2) adding a configurable retention interval (`VLLM_PREFIX_CACHE_RETENTION_INTERVAL`) to sparsify sliding-window checkpoints, reducing memory pressure; (3) updating the free logic in `remove_skipped_blocks` and `free` to separate cached and uncached blocks; (4) adding validation and tests for the new feature. The motivation is that DeepSeek V4's effective prefix-cache capacity was much lower than expected due to sliding-window allocations flushing cached blocks.

**Technical impact**  
The changes significantly improve prefix-cache hit rates for sliding-window attention models like DeepSeek V4. Microbenchmarks show hit rate increases from 0% to >95% for 14 concurrent 1M-context requests, matching the expected KV cache capacity. The new retention interval allows users to trade off cache density for capacity, with `0` retaining only the latest replay boundary and positive integers retaining sparse checkpoints. The free queue reordering ensures that non-cached blocks are reused first, preserving cached blocks for longer. The changes are backward-compatible (default behavior unchanged when env var is unset).

**Potential risks**  
The new retention interval must be a multiple of the scheduler block size, otherwise an error is raised; this could break existing configurations if set incorrectly. The free queue reordering changes block eviction order, which might affect other models or workloads that rely on the previous FIFO behavior. The feature is only applied to sliding-window attention; other attention types (Mamba, linear) are explicitly excluded, which could be confusing if users expect uniform behavior. The validation logic raises an error if the env var is set on a model without sliding-window groups, which might break existing scripts that set the var unconditionally.

**Key insights**  
The root cause of low prefix-cache capacity was that sliding-window allocations flushed cached blocks from the free queue. By prioritizing reuse of non-cached blocks, the fix preserves cached blocks without increasing total memory. The selective retention interval further reduces memory pressure by only caching checkpoints at configurable intervals, which is especially useful for long-context models. The separation of cached and uncached blocks in free operations is a clean design that could be extended to other cache types in the future.
