### Summary

**What changed and why**  
Added a new compile-time fusion pattern (RopeStaticQQuantKVCachePattern) that fuses RoPE, static Q FP8 quantization, and KV-cache update into a single fused operation. This improves decode performance for GPT-OSS style graphs by reducing intermediate operations and preserving dependency ordering. Also added corresponding tests and updated the fusion pass registration to include the new pattern.

**Technical impact**  
Introduces a new fusion pattern that combines RoPE, static FP8 quantization on Q, and KV-cache update. The pattern is registered before the existing generic RoPE+KV pattern and is guarded by capability checks for the static FP8 quant op and FP8 dtype. The fused op reuses the existing fused_rope_and_unified_kv_cache_update for the RoPE+KV portion, then applies static quantization to the output Q. **Potential risks**  
The new pattern is only applicable when static FP8 quant op is available and FP8 dtype is supported, so it won't affect other configurations. The pattern relies on the existing fused_rope_and_unified_kv_cache_update op, which must be correctly lowered. The test coverage includes a negative control to ensure the generic pattern does not match this graph, but there is a risk of unintended interactions if the pattern matching is too broad. The change also modifies the uuid method to include the new pattern, which may affect caching behavior.

**Key insights**  
The fusion is achieved by keeping the RoPE+KV-cache update fused and then applying static quantization to the Q output, rather than fusing quantization into the op itself. This preserves the explicit dependency ordering required by attention. The pattern uses auto_functionalized to handle in-place operations correctly. The test includes a negative control that disables the static Q fusion capability to verify the generic pattern cannot match this graph, ensuring the new pattern is solely responsible for the fusion.

  - `vllm/v1/spec_decode/llm_base_proposer.py`
  - `vllm/v1/worker/gpu_model_runner.py`

### Summary

**What changed and why**  
Extended the existing --use-fp64-gumbel flag to cover V1 sampling paths that use the exponential-race form of Gumbel-max sampling (q.exponential_(); probs / q; argmax). Previously, only the explicit Triton Gumbel sampler used fp64; V1 top-k/top-p sampling, spec decode recovery, and LLM draft proposer sampling still used fp32, which truncates the lower tail of exponential noise and can systematically miss rare-token wins in many-tail races. This change threads use_fp64_gumbel through the Python V1 sampler, TopKTopPSampler, rejection sampler, and LLM draft proposer to draw Exp(1) noise in float64 and compute race scores in float64 when enabled, while preserving the existing fp32 fast path by default.

**Technical impact**  
Adds ~391 lines and modifies 11 files. The core changes are in topk_topp_sampler.py (new helper functions empty_exponential_noise_like and sample_with_exponential_noise, updated random_sample and forward methods), rejection_sampler.py (threads use_fp64_gumbel through sample_recovered_tokens and the Triton kernel), llm_base_proposer.py (uses the new helpers in compute_probs_and_sample_next_token), and sampler.py/gpu_model_runner.py (wires the flag from ModelConfig into Sampler and TopKTopPSampler). The config docstring is updated. A CUDA proof script and targeted unit tests are added. Performance impact: fp64 paths are slower but only activated when use_fp64_gumbel=True; default behavior is unchanged.

**Potential risks**  
Risk of regression in the default fp32 path is low because the changes are guarded by the use_fp64_gumbel flag. The proof script is not a unit test and may not run on all environments.

**Key insights**  
The exponential-race form of Gumbel-max sampling is mathematically equivalent to the explicit Gumbel-max, so the same fp32 tail-truncation issue applies. The provided proof script demonstrates that fp32 exponential noise has a lower-tail cutoff around 2^-24, and in a many-tail race (100k trials, 262k tail tokens, gap 20.5), fp32 produces 0 tail hits while fp64 produces the expected ~32 hits. This change ensures that users who enable use_fp64_gumbel get consistent behavior across all sampling paths.
