## 2. [[Inductor] Fast-path Inductor fallback for vllm::*/vllm_aiter::* custom ops](https://github.com/vllm-project/vllm/pull/42129)


### Base Information

- **PR Number:** #42129
- **Author:** [okorzh-amd](https://github.com/okorzh-amd)
- **Merged By:** [AndreasKaratzas](https://github.com/AndreasKaratzas)
- **Merged time:** 2026-06-03 22:03:53
- **Type:** `SUBSTANTIAL`
- **[Changed Files](https://github.com/vllm-project/vllm/pull/42129/files) (2):**
  - `tests/compile/test_inductor_fallback_allow_list_patch.py`
  - `vllm/env_override.py`

### Summary

**What changed and why**  
Added a thin proxy class `_VllmFallbackAllowList` that wraps `torch._inductor.lowering.FALLBACK_ALLOW_LIST` to automatically treat any custom op in the `vllm::` or `vllm_aiter::` namespaces as a member. This avoids the expensive `operator_str` log formatting in `GraphLowering.call_function` which recursively stringifies deep IR provenance trees, causing minutes-long hangs during `torch.compile` for large MoE/TP graphs (e.g., Kimi-K2.6 at TP=8). The patch is applied at import time in `env_override.py` and is idempotent.

**Technical impact**  
Compile-time performance for large MoE/TP graphs is drastically improved: the per-op O(graph-depth) stringification is skipped for all `vllm::*` and `vllm_aiter::*` custom ops, reducing compile time from many minutes per op to near-instant. No runtime behavior changes; fallback semantics, pattern-matching passes (e.g., `allreduce_rms_fusion`), and partitioning remain identical. The change only affects the compile path and only for the specified namespaces.

**Potential risks**  
The proxy delegates `__contains__` for non-vllm strings to the underlying set, preserving existing behavior. However, any code that relies on `FALLBACK_ALLOW_LIST` being exactly an `OrderedSet` (e.g., type checks or direct attribute access not forwarded by `__getattr__`) could break. The `__getattr__` fallback should cover most cases, but edge cases like `isinstance` checks or `copy.copy` might fail. Additionally, if upstream Inductor changes the import structure or the `FALLBACK_ALLOW_LIST` type, the patch may need updates.

**Key insights**  
The root cause was not a missing lowering but an expensive log formatting path triggered on the first encounter of an unregistered custom op. By short-circuiting the membership check for vllm namespaces, the fix avoids the O(graph-depth) stringification without altering any fallback logic. The proxy pattern ensures backward compatibility with existing Inductor code that mutates or iterates the allow list.

**Key insights**  
Zero-copy RDMA is achieved by registering all expert weights and expert_buffer with NIXL at init, then issuing one-sided READs in add_recv using pre-computed source row mappings.   - `vllm/models/deepseek_v4/nvidia/ops/sparse_attn_compress_cutedsl.py`

### Summary

**What changed and why**  
Replaced the generic C128 split compressor kernel with a specialized `SparseAttnCompressC128Block8Kernel` optimized for the DeepSeek V4 C128 layout (head_size=512, state_width=512, compress_ratio=128, state-cache block size=8). The new kernel avoids dynamic block lookup and offset calculation by directly loading two consecutive state blocks per warp, reducing indexing overhead and improving memory access efficiency. C4 path remains unchanged using the fused kernel.

**Technical impact**  
Performance improvement of 1.08x to 1.69x across token sizes 1 to 16374 on B300, with larger gains at higher token counts due to better utilization of the specialized layout. The change also adds compile-time static KV-cache block size and validation that `kv_cache_block_size == k_cache.shape[1]`, and rejects unsupported C128 layouts early to prevent silent incorrect results.

**Potential risks**  
The kernel is intentionally scoped to a specific layout; any deviation (e.g., different head_size, state_width, compress_ratio, or block size) will be rejected. This could break if future model versions change these parameters. The C4 path is unchanged, so no risk there. The new kernel uses `shuffle_sync` for broadcasting, which is correct but relies on warp-level synchronization; any future compiler changes could affect this.

**Key insights**  
Specializing for the exact production layout yields significant speedups (up to 1.69x) by eliminating dynamic computations. The design uses 8 warps each processing 16 rows, with two state blocks loaded per warp, reducing global memory transactions. The PR also improves code clarity by removing generic parameters and adding explicit validation.
