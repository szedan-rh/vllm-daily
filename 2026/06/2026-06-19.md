## 1. [[Bugfix] Fix gridDim.y overflow for large row counts](https://github.com/vllm-project/vllm/pull/45255)


### Base Information

- **PR Number:** #45255
- **Author:** [JasonLi314](https://github.com/JasonLi314)
- **Merged By:** [tlrmchlsmth](https://github.com/tlrmchlsmth)

- **Type:** `SUBSTANTIAL`
- **[Changed Files](https://github.com/vllm-project/vllm/pull/45255/files) (2):**
  - `csrc/libtorch_stable/quantization/w8a8/fp8/per_token_group_quant.cu`
  - `tests/kernels/quantization/test_per_token_group_quant.py`

### Summary

**What changed and why**  
Fixed a bug in per_token_group_quant_8bit_packed kernel launch where gridDim.y was used for the row dimension (mn), which can exceed CUDA's 65535 limit for gridDim.y. The fix swaps gridDim.x and gridDim.y so that the row dimension uses gridDim.x (limit 2^31-1) and the scale factor dimension uses gridDim.y. Also updated the launch guard to check gridDim.y against 65535 instead of INT32_MAX. Added a regression test with mn=65537 to reproduce the error.

**Technical impact**  
Resolves a CUDA invalid configuration argument error when the number of rows (mn) exceeds 65535, which can occur in scenarios like DeepSeek-V4 MTP draft path with large token counts. The fix ensures correct kernel launch parameters for large row counts without changing the kernel logic.

**Potential risks**  
Low risk: the change only swaps grid dimensions and updates the guard check. The kernel logic remains unchanged. The new test verifies correctness against a reference implementation. Potential risk if other code depends on the previous grid dimension ordering, but the kernel is only called from the modified launch function.

**Key insights**  
CUDA gridDim.y is limited to 65535, while gridDim.x can be up to 2^31-1. The original code incorrectly assumed both dimensions had the same limit and mapped the potentially large row dimension to gridDim.y. This bug was latent because typical token counts are small, but surfaced with large batch sizes or profile runs. The fix is a simple swap of grid dimensions and adjustment of the guard condition.

## 4. [[Bugfix][Parser] Fix U+FFFD leak at reasoning-to-content transition in engine parsers](https://github.com/vllm-project/vllm/pull/46159)


### Base Information

- **PR Number:** #46159
- **Author:** [bbrowning](https://github.com/bbrowning)
- **Merged By:** [sfeng33](https://github.com/sfeng33)
- **Merged time:** 2026-06-19 15:09:29
- **Type:** `SUBSTANTIAL`
- **[Changed Files](https://github.com/vllm-project/vllm/pull/46159/files) (5):**
  - `tests/parser/engine/replay_harness.py`
  - `tests/parser/engine/test_delegating_replay.py`
  - `tests/parser/engine/test_ufffd_reasoning_transition.py`
  - `vllm/parser/abstract_parser.py`
  - `vllm/parser/engine/parser_engine.py`

### Summary

**What changed and why**  
Fixed a bug where U+FFFD (Unicode replacement character) leaked into streamed content at the reasoning-to-content transition. The old code decoded isolated content-side token IDs via tokenizer.decode(), which produced U+FFFD when byte-fallback tokens spanned the boundary. The fix flushes the reasoning parser's engine lexer instead, using finish_streaming() to get the complete text. Also added a regression test and refactored CHUNK_SIZES into a shared constant.

**Technical impact**  
Eliminates spurious U+FFFD characters in streaming output for models with reasoning parsers (e.g., GLM-4.7, Qwen3). The fix changes the delta message construction logic in abstract_parser.py and adds a finish_streaming() method in parser_engine.py that includes deferred content. The change is backward-compatible for non-engine-based parsers.

**Potential risks**  
Low risk. The change is localized to the reasoning-to-content transition path and is guarded by the _engine_based flag. The new finish_streaming() method may return a delta message even when no events are present (if deferred content exists), which could affect callers that previously expected None. However, the only caller (abstract_parser.py) handles this correctly.

**Key insights**  
The root cause was that byte-fallback tokens (e.g., multi-byte UTF-8 characters split across token boundaries) decode to U+FFFD when decoded in isolation. Flushing the engine lexer ensures the complete byte sequence is processed. The regression test uses a mock tokenizer that reproduces the U+FFFD behavior and validates the fix across multiple chunk sizes and parser configurations.

## 10. [[DSv4] Pack KV caches into contiguous per-block allocations for DeepSeek V4](https://github.com/vllm-project/vllm/pull/44577)


### Base Information

- **PR Number:** #44577
- **Author:** [tlrmchlsmth](https://github.com/tlrmchlsmth)
- **Merged By:** [tlrmchlsmth](https://github.com/tlrmchlsmth)
- **Merged time:** 2026-06-19 09:55:42
- **Type:** `SUBSTANTIAL`
- **[Changed Files](https://github.com/vllm-project/vllm/pull/44577/files) (7):**
  - `tests/v1/core/test_contiguous_kv_packing.py`
  - `vllm/distributed/kv_transfer/kv_connector/v1/nixl/base_worker.py`
  - `vllm/distributed/kv_transfer/kv_connector/v1/offloading/worker.py`
  - `vllm/v1/core/kv_cache_utils.py`
  - `vllm/v1/kv_cache_interface.py`
  - `vllm/v1/worker/gpu/attn_utils.py`
  - `vllm/v1/worker/gpu_model_runner.py`

### Summary

**What changed and why**  
For DeepSeek V4, pack all layer KV cache data contiguously per block so that KV connectors can send/receive one region per block instead of many tiny regions. Previously, each layer had its own allocation, resulting in 92 NIXL regions and 92 tiny P2P transfers per block (~16KB each). Now, all layers share one contiguous allocation per block, reducing to 1 region and 1 large RDMA transfer per block (~1.48MB). This is achieved by allocating a single backing tensor and creating strided views per layer with appropriate offsets and block strides.

**Technical impact**  
Reduces NIXL region count from 92 to 1 and P2P transfers from 92 to 1 per block, significantly lowering overhead and improving RDMA efficiency. The change modifies KV cache allocation, reshaping, and registration logic across multiple files, including kv_cache_utils, kv_cache_interface, attn_utils, gpu_model_runner, and the NIXL connector. New fields (offset, block_stride) are added to KVCacheTensor to describe the packed layout. The offloading worker also updates its tensor view creation to handle non-zero storage offsets.

**Potential risks**  
The packing assumes all layers in a group share the same block stride and are interleaved per block. If layer page sizes vary, the packing may waste memory or require padding. The detection of packed allocation in register_kv_caches relies on checking storage pointers, which could be fragile if tensors share storage but are not packed. The reshaping logic uses strided views that may not be compatible with all attention backends or future changes. Tests cover basic packing properties but may not cover edge cases like mixed page sizes or non-MLA layers.

**Key insights**  
The change optimizes KV cache transfer by exploiting the per-block interleaving pattern of DeepSeek V4, reducing RDMA overhead by two orders of magnitude. The design uses a single backing tensor with strided views, which is a common pattern for memory-efficient packing. The detection heuristic (same storage, different data pointers) is clever but should be documented more clearly. The test file provides good coverage of packing invariants (block stride, offsets, independence of views).
