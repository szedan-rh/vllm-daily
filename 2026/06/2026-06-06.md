### Base Information

- **PR Number:** #44560
- **Author:** [njhill](https://github.com/njhill)

- **Merged time:** 2026-06-06 16:05:48

  - `tests/v1/e2e/general/test_mamba_prefix_cache.py`
  - `tests/v1/kv_connector/unit/test_remote_prefill_lifecycle.py`
  - `tests/v1/kv_connector/unit/utils.py`
  - `vllm/v1/core/kv_cache_manager.py`
  - `vllm/v1/core/sched/scheduler.py`

### Summary

**What changed and why**  
This PR fixes a deadlock in async KV loading where multiple requests could each hold prefix blocks and then be unable to complete their full sequence due to insufficient free blocks. It introduces a reservation-based admission control: before admitting an async KV load, the scheduler computes how many blocks are reserved for already in-flight prefills (including other async loads) and only admits the new load if it fits within the remaining free blocks. This prevents the wedge condition without requiring preemption.

**Technical impact**  
Adds a `reserved_blocks` parameter to `allocate_slots` and a new `_inflight_prefills` set in the scheduler to track requests that are still prefilling (either via chunked prefill or async KV load). The `_inflight_prefill_reserved_blocks` method sums the remaining block needs of these requests, and this reservation is passed to `allocate_slots` for async loads. The change is backward-compatible (default `reserved_blocks=0`). Tests verify the deadlock scenario is resolved and that the gate is not over-conservative when pool is large.

**Potential risks**  
The reservation calculation assumes `_request_remaining_blocks` accurately reflects future allocations; if the block allocation logic changes, the reservation may become stale. The `_inflight_prefills` set must be correctly maintained across preemptions and request completions; any missed discard could cause false throttling. The change only affects async KV loads, so chunked prefills without async loading are unaffected.

**Key insights**  
The fix is simpler and more efficient than the previous lateral-preemption approach (PR #40968) because it avoids preemption entirely by preventing the deadlock condition at admission time. The reservation gate naturally throttles async loads without affecting chunked prefills, which are already throttled by batch capacity and preemptible. The approach is validated by both unit and end-to-end tests.
