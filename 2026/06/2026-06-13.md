**Key insights**  
The PR addresses a systematic inconsistency between build-time arch selection and runtime support checks. The approach of removing +PTX from global lists and requiring local addition is a cleaner design. The use of CUDART_VERSION instead of CUDA_VERSION for runtime header guards is more correct. ## 5. [[Perf] SM90 cutlass fp8 mm supports odd M by swap_ab, 180~290% kernel performance improvement](https://github.com/vllm-project/vllm/pull/44572)


### Base Information

- **PR Number:** #44572
- **Author:** [yewentao256](https://github.com/yewentao256)
- **Merged By:** [vllm-bot](https://github.com/vllm-bot)
- **Merged time:** 2026-06-13 12:05:45
- **Type:** `SUBSTANTIAL`
- **[Changed Files](https://github.com/vllm-project/vllm/pull/44572/files) (3):**
  - `csrc/libtorch_stable/quantization/w8a8/cutlass/c3x/scaled_mm_blockwise_sm90_fp8_dispatch.cuh`
  - `tests/kernels/quantization/test_cutlass_scaled_mm.py`
  - `vllm/model_executor/kernels/linear/scaled_mm/cutlass.py`

### Summary

**What changed and why**  
This PR improves performance of FP8 block-scaled matrix multiplication on SM90 (Hopper) GPUs when the M dimension is odd (not divisible by 4). Previously, odd M values required padding the input to a multiple of 4, which wasted memory and compute. The change introduces a swap_ab flag in the cutlass kernel template that transposes the problem (swapping A and B roles) when M is odd, allowing the kernel to handle odd M natively without padding. The dispatch logic in the CUDA kernel and the Python wrapper are updated to select the swapped kernel configuration when M % 4 != 0. The old padding-based fallback code is removed.

**Technical impact**  
Kernel performance for odd M improves by 180-290% (1.87x to 2.88x speedup) as shown in benchmarks. The change removes the need for padding tensors and copying data, reducing memory overhead and latency. The kernel now supports arbitrary M values without restrictions, simplifying the API and removing the m % 4 == 0 check. The swap_ab technique is adapted from SM100/SM120 architectures, leveraging existing cutlass infrastructure.

**Potential risks**  
The swap_ab path uses different tile shapes (128x16x128 vs 128x128x128) and a different epilogue scheduler (TmaWarpSpecialized vs TmaWarpSpecializedCooperative) and mainloop scheduler (Pingpong vs Cooperative). This may have different numerical behavior or performance characteristics for some problem sizes. The change assumes that swapping A and B is always beneficial for odd M, but for very small M or specific shapes, the overhead of the swap might not be worth it. The unit test modification removes a skip for odd M on SM100, which could expose previously untested behavior. The removal of the Python padding code and custom ops may affect any code that relied on those ops directly.

**Key insights**  
The swap_ab technique is a clever way to handle odd M dimensions by transposing the matrix multiplication problem, effectively making the odd dimension become N instead of M, which the cutlass kernel can handle natively. The performance improvement is significant (up to 2.88x) because it avoids the overhead of padding and copying tensors. The implementation reuses existing cutlass infrastructure for SM100/SM120, demonstrating cross-architecture code reuse. The change also simplifies the codebase by removing the padding logic and custom ops, reducing maintenance burden.
