Modified the Mooncake store worker to disable PUT striping when decode context parallelism (DCP) > 1. Previously, when num_kv_head < tp_size, TP ranks sharing KV heads striped PUTs across a single key namespace, which is invalid under DCP because each rank GETs from its own @dcpN namespace, causing OBJECT_NOT_FOUND errors. The fix adds a condition `self.dcp_size <= 1` to the striping logic, and adds a comprehensive unit test parametrized over dcp_size to verify the invariant that every key a rank GETs was PUT by some rank.

Fixes a correctness bug in Mooncake KV offloading when DCP > 1, which previously caused ~75% key retrieval failures (OBJECT_NOT_FOUND) at typical configurations (TP=DCP=4 with MLA). No other code paths are modified. The fix is logically sound: DCP splits the TP group into separate namespaces, so striping across ranks in different namespaces is incorrect.

**Key insights**  
The root cause is a subtle interaction between TP-level deduplication (striping PUTs across ranks sharing KV heads) and DCP's namespace partitioning. The fix correctly identifies that DCP creates separate @dcpN namespaces per rank, so striping must be disabled. PCP does not require a guard because it is an outer axis relative to TP. The test design is excellent: it directly asserts the invariant that every GET key must have a corresponding PUT, which is the exact failure mode observed.

  - `vllm/v1/worker/gpu/sample/gumbel.py`

### Summary

**What changed and why**  
Improved FP32 Gumbel sampling accuracy by replacing `log(u)` with `log1p(-u)` in the noise generation, which keeps the critical tail region in the well-resolved fp32 range near zero instead of near one. Added comprehensive tests (227 lines) covering accuracy, edge cases, and distribution fidelity.

**Technical impact**  
The FP32 Gumbel noise tail is now more accurate, allowing the sampler to correctly handle heavy-tailed distributions where the winning token can be many logits below the top. The change uses `tldevice.log1p` to avoid precision loss when computing `1-u` for small u. Tests verify that the empirical distribution matches the target within statistical tolerance, including for large vocabularies and non-block-aligned sizes.

**Potential risks**  
The `log1p(-u)` approach requires that `u` is never exactly 1 (which would cause `log1p(-1) = -inf`), but `tl.rand` returns values in [0, 1) so this is safe. The new constant `_TL_RAND_MIN` is the smallest positive value from `tl.rand` (4.6566127342e-10), which is larger than the previous `_FP32_TINY` (1.17549435e-38), so clamping is less aggressive but still prevents `log(0)`. Performance impact is negligible as the change is a single arithmetic operation.

**Key insights**  
The key insight is that fp32 has finer resolution near zero than near one, so moving the Gumbel noise tail computation to the region near zero via `log1p(-u)` significantly improves accuracy without changing the distribution. The heavy-tailed test design specifically exercises the noise tail, making the accuracy improvement measurable. The new tests also cover edge cases like zero-temperature greedy sampling, zero-count tokens, and non-block-aligned vocab sizes.
