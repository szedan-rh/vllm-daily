- **[Changed Files](https://github.com/vllm-project/vllm/pull/44665/files) (1):**

### Summary

**What changed and why**  
Changed the dtype of src_ptrs and dst_ptrs buffers from torch.int64 to torch.uint64 in Mamba state buffer creation to fix pointer overflow when device memory addresses exceed signed int64 maximum (2^63-1).

**Technical impact**  
Enables support for full 64-bit unsigned address space (up to 2^64-1) for device pointers, preventing overflow errors on systems with high memory addresses. This is a low-risk change as it only affects the buffer type used for pointer storage.

**Potential risks**  
Minimal risk; uint64 is the correct type for addresses. However, any downstream code that assumes signed int64 for these buffers may need adjustment, though unlikely given the buffers are internal to Mamba operations.

**Key insights**  
The fix addresses a real-world issue where device memory can exceed 2^63-1, which is possible with large GPU memory configurations. Using unsigned integers for pointer arithmetic is standard practice in low-level memory management.

## 2. [[Bugfix][Model Runner V2] Fix min_tokens off-by-one in the V2 GPU sampler](https://github.com/vllm-project/vllm/pull/46243)


### Base Information

- **PR Number:** #46243
- **Author:** [Sunt-ing](https://github.com/Sunt-ing)
- **Merged By:** [njhill](https://github.com/njhill)
- **Merged time:** 2026-06-20 22:06:50
- **Type:** `SUBSTANTIAL`
- **[Changed Files](https://github.com/vllm-project/vllm/pull/46243/files) (1):**
  - `vllm/v1/worker/gpu/sample/logit_bias.py`

### Summary

**What changed and why**  
Fixed off-by-one error in V2 GPU sampler's min_tokens logic. The condition `pos < min_len` was changed to `pos + 1 < min_len` because `pos` is the index of the last token (0-based), so comparing it directly against `min_len` (which is a count) delays EOS unblocking by one step. The fix aligns behavior with V1's MinTokensLogitsProcessor.

**Technical impact**  
Corrects the min_tokens parameter for V2 model runner on GPU, ensuring that EOS token is allowed exactly at position min_tokens (the (min_tokens+1)-th token) instead of one step later. This affects all models using the V2 GPU sampler (e.g., Llama, Qwen3, Mistral) when min_tokens > 0.

**Potential risks**  
Low risk. The change is a single-line fix with a clear semantic correction. The condition now correctly uses 1-based length comparison. The guard `num_stop_token_ids > 0` ensures min_tokens=0 is unaffected. Thorough testing against V1 reference confirms exact match.

**Key insights**  
The bug was a classic off-by-one error: using a 0-based position index as if it were a 1-based count. The fix is minimal and well-validated with a forced-EOS test across multiple min_tokens values. The PR description includes a clear test plan and results table, demonstrating correctness.

**Technical impact**

**Key insights**
