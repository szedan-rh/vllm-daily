### Summary

**What changed and why**  
Added `--index-strategy unsafe-best-match` flag to `uv pip install` commands in CPU installation docs for both ARM and x86. This fixes a dependency resolution failure where `uv` could not find the required `setuptools` version because it only checked the PyTorch index first and did not fall back to PyPI.

**Technical impact**  
Users following the CPU source build instructions will no longer encounter a `uv` dependency resolution error for `setuptools`. The flag allows `uv` to consider package versions from all indexes, resolving the conflict between the PyTorch index (older setuptools) and PyPI (newer setuptools).

**Potential risks**  
The `unsafe-best-match` strategy reduces protection against dependency confusion attacks by considering packages from all indexes equally. However, the risk is low because the indexes used are official (PyTorch and PyPI). Users should ensure they trust the configured indexes.

**Key insights**  
The issue stems from `uv`'s default strict index strategy, which prioritizes the first index containing a package. This is a common pitfall when using multiple package indexes. The fix is a simple flag addition that aligns with `uv`'s documentation for multi-index scenarios.

### Summary

**What changed and why**  
Added two new endpoints POST /v1/chat/completions/derender and POST /v1/completions/derender to the render server, completing the render-to-derender lifecycle for disaggregated serving. The generate server returns raw token IDs and logprob placeholders (token_id:N) because it lacks a tokenizer; these new endpoints detokenize output tokens, resolve logprob placeholders into real token strings with bytes fields, and format the result as standard ChatCompletionResponse or CompletionResponse. Also added helper functions format_token_id_placeholder and resolve_token_id_placeholder, updated existing logprob creation to use the helper, and added request models DerenderChatRequest and DerenderCompletionRequest.

The new endpoints depend on the tokenizer being available on the render server; if the tokenizer is missing or incompatible, derendering will fail. The resolve_token_id_placeholder function assumes the token_id:N format and may silently return the original token if the format is unexpected, which could mask errors. The completion derender endpoint aggregates kv_transfer_params across multiple responses and warns if they differ, but sets them to None, which may break downstream consumers expecting consistent parameters. The new endpoints are not yet covered by integration tests beyond the unit tests provided.

**Key insights**  
This PR completes the disaggregated serving lifecycle by adding the postprocessing step on the CPU-only render server, enabling a clean separation where the GPU generate server only handles token generation without a tokenizer. The design is stateless and idempotent, making it suitable for scaling. The inclusion of the original request objects (chat_request/completion_request) in the derender request models anticipates future parsing needs without requiring protocol changes later.

### Summary

**What changed and why**  
Modified the condition in `_free_encoder_inputs` to subtract `num_output_placeholders` from `num_computed_tokens` before comparing with the end of the encoder placeholder range. This prevents premature freeing of encoder cache entries when speculative decoding is used, because optimistic advancement of `num_computed_tokens` can later be rolled back by draft token rejection. Added four unit tests covering the rollback path, non-speculative behavior, preemption retention, and eviction during preemption.

**Technical impact**  
Fixes a bug where the scheduler could free encoder cache entries that are still needed after a rollback in speculative decoding, causing 'Encoder cache miss' crashes. The change is minimal (one line modified) but critical for correctness in async scheduling with speculation. The new tests ensure regression coverage and verify that the fix does not affect non-speculative paths.

**Potential risks**  
Low risk: the change only affects the condition for freeing encoder inputs when `num_output_placeholders > 0`. Non-speculative paths are unchanged. The subtraction could theoretically delay freeing slightly, but only by the number of unconfirmed draft tokens, which is bounded and small. The tests cover the key scenarios.

**Key insights**  
The root cause is that `num_computed_tokens` can be optimistically advanced past the encoder range end, but later rolled back by draft token rejection. The fix uses the confirmed progress (`num_computed_tokens - num_output_placeholders`) to decide when it is safe to free. The preemption path is handled separately by the encoder cache manager's freeable list, and the tests confirm that the scheduler does not report freed hashes prematurely.

### Summary

**What changed and why**  
Fused the inverse GPT-J RoPE operation into a single Triton kernel and cached the dequantized bf16 wo_a weight to avoid per-step dequantization. This addresses performance bottlenecks identified in ROCm DeepSeek V4 decode profiling, where multiple small PyTorch kernels and repeated fp8-to-bf16 conversions were the largest copy/mul operations.

**Technical impact**  
Replaces ~10 small PyTorch kernels (clone, index_select, repeat_interleave, neg, stack, cat, casts) with a single Triton kernel for inverse RoPE, and caches the bf16 wo_a weight on the module so dequantization runs once instead of every decode step. Achieves 8.9%–15.6% throughput gain and 7.9%–13.3% TPOT reduction across concurrency levels 4–64. Unit tests confirm bitwise match for NoPE lanes and close tolerance for RoPE lanes.

**Potential risks**  
The Triton kernel assumes contiguous last dimension and even rope_dim; fallback PyTorch path is kept for non-conforming inputs. Caching the dequantized weight increases memory usage by storing a bf16 copy of wo_a, but this is static and shared across decode steps. The change is guarded by assertions and falls back gracefully if conditions are not met.

**Key insights**  
The fusion and caching directly address the two largest kernel costs in the decode profile, demonstrating that targeted kernel fusion and weight caching can yield significant performance gains (up to 15.6% throughput) without changing the model's numerical behavior. The approach mirrors SGLang/ATOM's strategy of keeping wo_a in bf16 for plain bf16 GEMM.
