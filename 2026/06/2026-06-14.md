  - `rust/src/llm/src/request_metrics.rs`
  - `rust/src/llm/tests/generate.rs`
  - `rust/src/text/src/lib.rs`

### Summary

**What changed and why**  
Added an external→internal request-id mapping (InflightRequests) to the Rust frontend's Llm, TextLlm, and ChatLlm facades, enabling abort() by user-supplied request ids. The mapping tracks which internal engine request ids correspond to each external id, with refcounted edges and RAII guards for cleanup. Also added local abort finalization in EngineCoreClient to synthesize terminal Abort outputs for client-initiated aborts, and fixed request metrics to skip token-timing for empty token outputs (e.g., abort terminals).

**Technical impact**  
Introduces a new inflight.rs module with a concurrent HashMap protected by parking_lot::Mutex, and a RequestGuard that decrements refcounts on drop via a Weak reference. The Llm::generate() method now tracks the external→internal edge after a successful client.call(), and Llm::abort() resolves external ids to internal ids before calling client.abort(). The EngineCoreClient's abort path now locally finalizes streams with a synthesized Abort output before sending the abort to engines. The request_metrics tracker skips token-timing for outputs with empty new_token_ids to avoid spurious metrics on abort terminals.

**Potential risks**  
The InflightRequests map uses a Mutex that could become a contention point under high concurrency, though the critical sections are short. The Weak reference in RequestGuard means if the InflightRequests is dropped before all guards, drops are no-ops, which is safe but could hide bugs if guards are held longer than expected. The local abort finalization in EngineCoreClient adds a new code path that must be kept in sync with engine-side abort handling. The change to request_metrics may skip legitimate token-timing for outputs that happen to have empty token_ids but are not aborts (though such outputs are unlikely).

**Key insights**  
The design mirrors Python's AsyncLLM.abort() by resolving external ids to internal ids at the facade layer, keeping engine-core unaware of external ids. The refcounted edges handle the edge case where request-id randomization is disabled and the same (external, internal) pair is reused. The local abort finalization ensures streams terminate cleanly even if the engine is unreachable, improving robustness. The test coverage includes unit tests for the inflight map and integration tests verifying correct internal id resolution and multi-request abort.

### Summary

**What changed and why**  
The Rust `EngineCoreReadyResponse` struct was missing three fields present in the Python dataclass: `block_size` (required), `kv_cache_size_tokens` (optional), and `kv_cache_max_concurrency` (optional). This caused a `msgspec.ValidationError` when the Python frontend decoded the response from a Rust engine. The fix adds these fields to the Rust struct and updates the mock engine's default response. A new Python fixture and Rust test assert that the field sets match.

**Technical impact**  
Restores compatibility between Rust engines and the Python frontend for the registration handshake. Without this fix, any Rust-based engine (including the mock engine) would fail to register, blocking all requests. The change also adds a regression test to prevent future drift.

**Potential risks**  
Low risk. The added fields are either required (block_size) or optional (kv_cache_*), so existing Rust engines that don't set them will now need to provide block_size. The mock engine and any production Rust engines must be updated accordingly. The test ensures field parity, but there is a minor risk of other fields drifting in the future if the Python dataclass changes again.

**Key insights**  
The bug was discovered by running a mock engine against the real Python frontend, highlighting the importance of end-to-end testing across language boundaries. The fix uses a Python fixture to generate a reference encoding and a Rust test to compare field keys, providing a robust mechanism to detect future mismatches.

Added a monkey-patch to prometheus-fastapi-instrumentator's internal route name resolution function to skip routes without a `.path` attribute. This fixes a regression introduced in FastAPI >=0.137 where lazy `_IncludedRouter` objects (which lack `.path`) cause AttributeError in the metrics middleware, resulting in 500 errors on every request.

**Technical impact**  
The patch modifies the behavior of `_get_route_name` in the instrumentator's routing module to check for the existence of `route.path` before accessing it. This ensures that path-less routes are skipped during metric label generation, preventing crashes while maintaining correct metric collection for all other routes. **Potential risks**  
The monkey-patch relies on internal implementation details of prometheus-fastapi-instrumentator (the `_get_route_name` function), which could change in future versions of that library. If the library updates its internal API, the patch may become ineffective or cause unexpected behavior. Additionally, skipping path-less routes means those routes will not contribute to request metrics, which is acceptable as they are internal lazy routes not serving actual requests.

**Key insights**  
The fix addresses a compatibility issue between vLLM and FastAPI 0.137+ by patching a third-party library's internal function. This approach is minimal and avoids forking the library, but it highlights the fragility of depending on private APIs. The patch is applied at module import time, ensuring it takes effect before any request is handled.

## 11. [[Bugfix][CPU] Honor cgroup memory limit when computing KV cache size](https://github.com/vllm-project/vllm/pull/45086)


### Base Information

- **PR Number:** #45086
- **Author:** [maobaolong](https://github.com/maobaolong)
- **Merged By:** [vllm-bot](https://github.com/vllm-bot)
- **Merged time:** 2026-06-14 19:26:51
- **Type:** `SUBSTANTIAL`
- **[Changed Files](https://github.com/vllm-project/vllm/pull/45086/files) (1):**
  - `vllm/utils/cpu_resource_utils.py`

### Summary

**What changed and why**  
Added cgroup memory limit awareness to `get_memory_node_info()` in `vllm/utils/cpu_resource_utils.py`. Previously, the function used host-wide NUMA memory numbers, causing KV cache allocation failures in containerized environments (Docker/K8s) where the cgroup limit is lower than host memory. Now it reads cgroup v1/v2 limits and clamps the reported total and available memory accordingly.

**Technical impact**  
Fixes a critical bug where vLLM would over-allocate CPU memory for KV cache inside containers, leading to `ValueError` and crashes. The change ensures that `gpu_memory_utilization` is applied to the container's actual memory quota rather than the host's total RAM. Bare-metal deployments are unaffected because the function returns `None` when no cgroup limit is found or the limit is effectively unlimited.

**Potential risks**  
Low risk. The change is guarded by `sys.platform == 'linux'` and only activates when a cgroup limit is present and smaller than the NUMA total. The cgroup limit is per-pod, not per-NUMA-node, so clamping is a reasonable approximation. Edge cases: if cgroup usage exceeds the limit (should not happen), `available_memory` could become negative but is clamped to 0. The `_read_int_file` helper handles missing files and invalid values gracefully.

**Key insights**  
The root cause was that `/sys/devices/system/node/node*/meminfo` reports host-wide memory, ignoring cgroup constraints. The fix reads `/sys/fs/cgroup/memory.max` (v2) or `/sys/fs/cgroup/memory/memory.limit_in_bytes` (v1) and clamps the NUMA totals. The v1 sentinel value (>= 2^62) is treated as unlimited. The change is minimal (+52 lines) and well-tested with manual verification in a cgroup-v1 container.
