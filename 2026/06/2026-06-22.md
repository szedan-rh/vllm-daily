**What changed and why**  
This PR introduces dedicated Tokio runtimes for HTTP request processing and ZMQ transport in the Rust frontend. The goal is to prevent CPU-heavy request preprocessing and engine transport work from blocking the HTTP worker threads, improving tail latency and throughput under high concurrency. A new request runtime offloads heavyweight routes (e.g., /v1/chat/completions) via a Tower middleware, and a ZMQ runtime handles engine-core communication. **Technical impact**

The request runtime is lazily initialized in AppState and used by a Tower middleware that offloads specific paths. The change also modifies the gRPC server setup, which may affect streaming behavior. Error handling for runtime task failures returns a generic 500 error, which might mask underlying issues.

**Key insights**  
The PR demonstrates a clear performance improvement by isolating CPU-bound work from I/O-bound HTTP handling. The use of a Tower middleware for offloading is a clean, non-invasive approach. The benchmark results are compelling, especially for health check tail latency (p99.9 reduced by 93%). The design is configurable via environment variables, allowing tuning for different workloads. The addition of a dedicated ZMQ runtime also improves isolation for engine communication.

**Technical impact**  
Introduces schema-aware streaming of string values: fields that can only be strings (per schema) are streamed immediately, while fields that may be coerced to other types still wait for stable serialization. **Potential risks**  
The streaming of unterminated string values could break if the schema is incorrect or if the model produces unexpected tokens. The new logic depends on correct schema extraction; if `find_tool_properties` returns incomplete properties, string keys might not be identified, falling back to old behavior. The whitespace stripping change in `_qwen3_arg_converter` could alter behavior for partial values that intentionally include leading/trailing whitespace. The new lexer terminals for PARAM_START and PARAM_END might interact with other transitions if not properly scoped.

**Key insights**  
The fix leverages the fact that string values are prefix-stable (unlike numbers or booleans) to reduce latency. The schema-aware approach ensures correctness for type-coerced fields. The Qwen3 parser change prevents partial tags from leaking into argument values by treating them as terminals. The test additions cover streaming long strings, split parameter tags, and whitespace handling, ensuring the new behavior is well-validated.

## 13. [[DeepEP V2] Bound num_max_tokens_per_rank in do_expand=False](https://github.com/vllm-project/vllm/pull/46404)


### Base Information

### Summary

**What changed and why**  
In DeepEPV2PrepareAndFinalize._do_dispatch, when do_expand=False (decode/cudagraph mode), num_max_tokens_per_rank was previously unset, causing the recv buffer to be sized to the worst-case R * max_num_batched_tokens. This change bounds it to the DP-padded batch size rounded up to a power of two, reducing decode step time significantly and preventing a JIT recompile storm.

**Technical impact**  
Decode step time improved 6.1× (100.7 ms → 16.4 ms) at concurrency=1, eliminated cicc recompile storm at concurrency=64, and improved throughput ~6% at concurrency=1024. Prefill (do_expand=True) is unaffected.

**Potential risks**  
The change relies on dp_metadata from forward context; if dp_meta is None, it falls back to tokens.shape[0]. The pow2 rounding may slightly over-allocate for non-power-of-two batch sizes, but this is bounded and intentional to limit kernel recompilations. No regression in accuracy (GSM8K unchanged).

**Key insights**  
The root cause was that an unset num_max_tokens_per_rank defaulted to the buffer's init capacity, causing massive over-allocation for decode. The pow2 bucketing is a clever trade-off to avoid JIT recompile storms while keeping buffer sizes small for decode.
