## 1. [[ROCm][gpt-oss] Pass GateMode.INTERLEAVE for MXFP4 W4A16 fused MoE](https://github.com/vllm-project/vllm/pull/44893)


### Base Information

- **PR Number:** #44893
- **Author:** [Rohan138](https://github.com/Rohan138)

  - `vllm/_aiter_ops.py`
  - `vllm/model_executor/layers/fused_moe/experts/rocm_aiter_moe.py`

### Summary

**What changed and why**  
Added a `gate_mode` parameter to the fused MoE call path, defaulting to empty string, and set it to `GateMode.INTERLEAVE` when `use_mxfp4_w4a16` is true. This fixes accuracy regressions and JIT crashes caused by a mismatch between interleaved weight shuffling (from `shuffle_weight_a16w4`) and the default separated gate mode in AITER's dispatcher after PR #3123.

**Technical impact**  
Enables correct MXFP4 W4A16 fused MoE execution on ROCm with AITER >=0.1.14, restoring accuracy from 0.0 to ~0.905 on gpt-oss-120b and preventing CK2stages JIT crashes on gpt-oss-20b-w-mxfp4-a-bf16. The change is backward-compatible: older AITER versions ignore the unknown `gate_mode` kwarg, and other quantization paths (W4A8, W8A8) are unaffected because they don't set `use_mxfp4_w4a16`.

**Potential risks**  
Low risk. The `gate_mode` parameter is only passed when `use_mxfp4_w4a16` is true and AITER supports it (probed via `fused_moe_supports_gate_mode`). Other paths remain unchanged. The import of `GateMode` is guarded by a try-except, so missing AITER versions won't crash. The only risk is if future AITER changes the `gate_mode` semantics, but that would be a separate compatibility issue.

**Key insights**  
The root cause was a silent behavioral change in AITER's dispatcher that made the previously tolerated mismatch between interleaved weights and separated gate mode now produce incorrect results or crashes. The fix aligns the gate mode with the actual weight layout, which is a straightforward one-line semantic correction. The validation matrix shows consistent accuracy recovery across multiple AITER versions and TP sizes.

### Summary

**What changed and why**  
Fixed a bug where CPU offloading with Mamba models in 'align' cache mode produced incorrect results when prompts ended exactly at offloaded block boundaries. The issue was that the offloading code treated MambaSpec like sliding-window attention, returning hit tokens not aligned to Mamba's single-state cache size. The fix rounds down the max hit tokens to the Mamba block size when a MambaSpec with 'align' mode is detected, preventing the CPU offload from loading cached blocks at boundaries.

**Technical impact**  
Adds a new function `resolve_mamba_align_size` to scan KV cache groups and determine the Mamba alignment size, and modifies the `_lookup` method to constrain the hit window to that size. A new unit test `test_mamba_align_cpu_offload` validates both block-boundary and mid-block scenarios. The change ensures correctness for Mamba models with CPU offloading and 'align' cache mode, matching cold-run outputs.

**Potential risks**  
The change is specific to MambaSpec with 'align' mode and only affects the offloading scheduler. It introduces an assertion that all Mamba groups agree on the same alignment size, which could fail if misconfigured. The test is CUDA-only and parametrized with a single model; broader coverage might be needed. No performance regression is expected as the rounding is a simple operation.

**Key insights**  
Mamba's single-state cache requires special handling at block boundaries during CPU offloading, unlike sliding-window attention which can reuse cached blocks. The fix ensures that when a prompt ends exactly at a block boundary, the offload does not load the cached block, avoiding incorrect recomputation. The test demonstrates that cold and CPU-hit outputs match after the fix.

**Key insights**  
Offloading CPU-bound preprocessing to a thread pool is a classic pattern to avoid blocking async I/O. The optimal thread count is workload-dependent: 2 threads gave best results for long audio, while 1 thread was slightly better for short audio. The PR includes careful benchmarking and a new utility function make_async_with_semaphore that combines executor offloading with concurrency limiting.
