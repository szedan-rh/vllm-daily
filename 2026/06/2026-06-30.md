  - `rust/src/server/src/routes/openai/completions/types.rs`

### Summary

**What changed and why**  
Added a `normalize()` method to `CompletionRequest` that coerces an explicit `null` `max_tokens` to the default value of 16. Previously, `null` was treated as 'no limit' causing runaway generation, while the documented default is 16. This matches Python vLLM's behavior.

**Technical impact**  
Fixes a bug where clients sending `{"max_tokens": null}` (e.g., OpenAI Python SDK) would get full-context generation instead of the default 16 tokens. The fix ensures consistent behavior between Rust and Python frontends.

**Potential risks**  
Low risk. The change only affects the case where `max_tokens` is explicitly `null`, which was previously broken. Existing requests with absent or valid `max_tokens` are unaffected. The `normalize()` method is already called post-deserialization, so no wiring changes are needed.

**Key insights**  
Serde's `#[serde(default)]` only applies when the key is missing, not when it's `null`. This subtlety caused the bug. The fix mirrors Python vLLM's `normalize_null_max_tokens` validator, ensuring cross-language consistency.

**Potential risks**  
The new tanh implementation is only available on ARM CPUs with NEON support; other architectures continue using PyTorch's native GELU tanh, so no regression there. The accuracy of the approximation is validated with unit tests (rtol/atol appropriate for each dtype). However, the change modifies the default behavior for ARM CPUs: previously they used PyTorch's GELU tanh, now they use the custom kernel. If the approximation error is not acceptable for some use cases, users may need to disable the custom op. Additionally, the new code adds complexity to the build system and platform configuration, which could introduce issues on non-standard ARM environments.

**Key insights**  
This PR demonstrates a targeted optimization for ARM CPUs by leveraging hardware-specific SIMD instructions (NEON) to accelerate a common activation function. The performance gains are substantial (up to 2.6x) and the implementation follows the Arm Optimized Routines pattern, ensuring numerical accuracy. The integration is clean: the new tanh function is added as a method to existing vectorized types, and the operator registration follows the existing pattern. The PR also includes comprehensive benchmarking and unit tests, showing careful validation.

Changes include: new CuteDSL-based stable-topk kernel for merging DCP candidates, Triton pack kernel, DCP-aware index conversion and filtering, localized sequence length computation, and LSE handling for decode. The indexer and metadata builders are extended to handle DCP-localized KV shards.

**Technical impact**  
Enables DCP for sparse MLA attention, reducing per-rank KV cache size proportionally to DCP world size (e.g., 4x reduction with DCP 4). New code paths are not yet validated for interleave > 1 or KV compression.

**Key insights**  
The DCP merge is exact (not approximate) because the global top-K must be a subset of the union of per-rank local top-Ks. The Triton pack kernel and CuteDSL selector are tightly coupled; no PyTorch fallback exists. The LSE normalization and empty-row masking are critical for correct DCP merge. The test suite includes a fake DCP group for unit testing without multi-process setup.
