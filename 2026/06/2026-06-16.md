## 10. [[M3] Tune Triton indexer score decode for spec-decode](https://github.com/vllm-project/vllm/pull/45743)


### Base Information

- **PR Number:** #45743
- **Author:** [gau-nernst](https://github.com/gau-nernst)
- **Merged By:** [zyongye](https://github.com/zyongye)
- **Merged time:** 2026-06-16 21:07:35
- **Type:** `SUBSTANTIAL`
- **[Changed Files](https://github.com/vllm-project/vllm/pull/45743/files) (3):**
  - `tests/kernels/attention/test_minimax_m3.py`
  - `vllm/models/minimax_m3/common/indexer.py`
  - `vllm/models/minimax_m3/common/ops/index_topk.py`

### Summary

**What changed and why**  
This PR optimizes the Triton indexer score decode kernel for spec-decode by batching multiple query tokens per CTA instead of launching one CTA per token. Previously, each CTA processed one query token across all heads; now each CTA processes a request's decode_query_len tokens across all heads, improving efficiency when spec-decode is active. The change also removes the sm_scale parameter (unnecessary for decode), reduces TARGET_GRID from 4096 to 512, and adds a max_decode_query_len constant to avoid recompilation. Test parameters are updated to cover various decode_query_len and max_decode_query_len combinations.

**Technical impact**  
The kernel launch grid changes from batch (flattened tokens) to seq_lens.shape[0] (number of requests), and the query tile expands to BLOCK_SIZE_Q = next_power_of_2(max_decode_query_len) across all heads. This increases register/memory usage per CTA but reduces total CTAs, improving occupancy and memory bandwidth utilization. Microbenchmarks show significant latency reductions (up to 48.9%) and bandwidth improvements (up to 95.7%) for spec-decode cases, with minimal regression for non-spec-decode. The change also removes sm_scale from the kernel, simplifying computation.

**Potential risks**  
The kernel now assumes decode_query_len <= max_decode_query_len, which is enforced by an assertion. If max_decode_query_len is set incorrectly, it could cause out-of-bounds memory access or incorrect results. The reduction in TARGET_GRID may affect performance for very large batch sizes, though the PR includes benchmarks showing improvements. The change to grid dimensions (from batch to seq_lens.shape[0]) could interact poorly with padding or zero-length requests; the code clamps query positions to handle padded rows. The removal of sm_scale is safe for decode because only block ordering is needed, but if the kernel is reused elsewhere, it may need adjustment.

**Key insights**  
The key insight is that for spec-decode, multiple query tokens per request can be processed together in a single CTA, improving efficiency by sharing K-block loads across queries. The use of max_decode_query_len as a constexpr avoids recompilation when switching between spec-decode and non-spec-decode modes. The tuning of num_warps and num_stages for the wide query tile (num_idx_heads > 1 and max_decode_query_len > 1) helps manage register pressure. The removal of sm_scale simplifies the kernel without affecting correctness since decode only uses block ordering.
