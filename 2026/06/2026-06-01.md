### Summary

**What changed and why**  
Updated three files to use the actual batch max_seq_len (instead of max_model_len) for attention metadata in V2 model runner paths. In default.py, removed unnecessary int() cast. Low risk; the change is consistent with the existing pattern in DefaultModelState. However, incorrect computation of draft_max_seq_len (e.g., if seq_lens_cpu_upper_bound is not properly updated) could lead to incorrect attention behavior. The min with max_model_len provides a safety cap.

**Key insights**  
This is a follow-up to PR #40654, extending the same fix to MambaHybrid and speculative decoding paths. The use of actual batch max_seq_len reduces unnecessary computation and prevents memory access violations. The draft model also accounts for future speculative steps by adding num_speculative_steps to the current max_seq_len.

## 6. [[BugFix][kv_offload]: Prevent offloading stale sliding window blocks](https://github.com/vllm-project/vllm/pull/42959)


### Base Information

- **PR Number:** #42959
- **Author:** [orozery](https://github.com/orozery)
- **Merged By:** [orozery](https://github.com/orozery)
- **Merged time:** 2026-06-01 19:59:49
- **Type:** `SUBSTANTIAL`
- **[Changed Files](https://github.com/vllm-project/vllm/pull/42959/files) (2):**
  - `tests/v1/kv_connector/unit/offloading_connector/test_scheduler.py`
  - `vllm/distributed/kv_transfer/kv_connector/v1/offloading/scheduler.py`

### Summary

**What changed and why**  
Fixed a bug where stale sliding window block IDs could be offloaded after a prepare_store failure. When prepare_store fails, offloading is delayed; meanwhile, sliding window blocks can be freed and reallocated to different logical blocks. Previously, this went undetected, causing wrongful KV data offloading or assertion errors. The fix adds tracking of newly allocated GPU block IDs per step and zeroes out stale block IDs in sliding window groups before building store jobs. It also flushes pending jobs that contain re-allocated blocks.

**Technical impact**  
Introduces a new set `_current_batch_allocated_block_ids` to track block IDs allocated in the current engine step. Adds `_update_req_states` method to zero out stale block IDs in sliding window groups. Moves flushing of pending jobs for re-allocated blocks from `update_state_after_alloc` and `_build_store_jobs` into `build_connector_meta`, consolidating the logic. The unit test `test_stale_sliding_window_block_after_prepare_store_failure` reproduces the bug and verifies the fix.

**Potential risks**  
The change modifies the order of operations in `build_connector_meta` and `_build_store_jobs`, which could affect other edge cases. The new logic assumes that only sliding window groups can have stale entries; if other groups can also have stale blocks, the fix may be incomplete. The flushing of pending jobs for re-allocated blocks now happens in `build_connector_meta` instead of earlier, which might delay flushing but should be safe. The unit test covers the specific scenario but may not cover all variations.

**Key insights**  
The root cause is that after a failed prepare_store, the offloading state retains stale block IDs that become invalid when blocks are freed and reallocated. The fix uses the set of newly allocated block IDs to detect and zero out stale entries before building store jobs. This prevents offloading of wrong data and avoids KeyError in `_remove_pending_job`. The approach is efficient because it only checks positions that could be stale (between next_stored_block_idx and the pre-extend length).

### Summary

**What changed and why**  
Moved the logic for flushing unstreamed tool-call arguments from the serving layer (OpenAIServingChat) into the parser layer (DelegatingParser._append_unstreamed_tool_args and ToolParser.get_remaining_unstreamed_args). This refactor centralizes streaming state management in the parser, reducing duplication and simplifying the serving code. The serving layer now passes a `finished` flag to `parse_delta` when `finish_reason` is set, allowing the parser to emit remaining arguments in its final delta. The `auto_tools_called` variable was removed as it is no longer needed.

**Technical impact**  
Reduces code complexity in the serving layer by removing ~120 lines of logic, including the `_create_remaining_args_delta` helper and the `_should_check_for_unstreamed_tool_arg_tokens` method. **Key insights**  
The refactor aligns with the principle of keeping streaming state within the parser, which is the natural owner of that state. The new `get_remaining_unstreamed_args` method uses a simple string prefix removal to compute unstreamed arguments, avoiding the previous double-serialization bug. The test coverage is improved with new unit tests for the parser's flush behavior, including edge cases where no flush is needed or where arguments are fully streamed. The overall change reduces coupling between the serving and parser layers.

### Summary

**What changed and why**  
Fixed GLM-5.1 FP8 model loading failure when using pipeline parallelism (pp=8). The issue was that the FP8 indexer WK weight loading logic did not account for layers not assigned to the current pipeline rank, causing a KeyError when trying to access missing parameters. Added a check using `get_pp_missing_layer_names` to skip loading FP8 indexer WK weights for layers that are not present on the current rank.

**Technical impact**  
Enables serving the GLM-5.1-FP8 model with pipeline parallelism up to 8 GPUs. The fix ensures that weight loading correctly handles distributed model sharding by ignoring weights for layers that belong to other pipeline stages. This is critical for scaling large models across multiple GPUs.

**Potential risks**  
Low risk. The change only affects the FP8 indexer WK weight loading path and adds a guard condition that returns early for missing layers. It does not alter the core model logic or other weight loading paths. The fix has been validated with a full GSM8K evaluation achieving high accuracy.

**Key insights**  
The bug was caused by the assumption that all layers' weights are present on every pipeline rank, which is false in pipeline parallelism. The solution leverages existing utility functions (`get_pp_missing_layer_names`) to identify layers not owned by the current rank and skip their FP8 indexer WK weights. This pattern can be applied to other similar weight loading issues in distributed settings.
