## 1. [[CPUOffloadingManager] Maintain evictable list in LRUCachePolicy](https://github.com/vllm-project/vllm/pull/46216)


### Base Information

- **PR Number:** #46216
- **Author:** [varun-sundar-rabindranath](https://github.com/varun-sundar-rabindranath)
- **Merged By:** [orozery](https://github.com/orozery)
- **Merged time:** 2026-06-21 23:54:45
- **Type:** `SUBSTANTIAL`
- **[Changed Files](https://github.com/vllm-project/vllm/pull/46216/files) (5):**
  - `tests/v1/kv_offload/cpu/test_manager.py`
  - `tests/v1/kv_offload/tiering/test_tiering_offloading.py`
  - `vllm/v1/kv_offload/cpu/manager.py`
  - `vllm/v1/kv_offload/cpu/policies/base.py`
  - `vllm/v1/kv_offload/cpu/policies/lru.py`

### Summary

**What changed and why**  
Added a dedicated evictable list (OrderedDict) to LRUCachePolicy to track blocks with ref_cnt == 0, enabling O(1) eviction instead of scanning all blocks. Updated insert, remove, touch, evict, and added mark_evictable/mark_non_evictable methods. Modified CPUOffloadingManager to call these methods when ref_cnt transitions. Updated tests to reflect new eviction order.

**Technical impact**  
Eviction in LRUCachePolicy is now O(n) where n is the number of evictable blocks (typically small) instead of O(N) where N is all cached blocks. This significantly reduces CPU overhead during eviction, as shown by throughput improvement from 0.1 req/s to 1.1 req/s in benchmarks. The change also affects LRU ordering: touch only moves evictable blocks, and active blocks are untouched until they become evictable.

**Potential risks**  
The eviction order may change because touch no longer moves active (non-evictable) blocks. This could alter cache behavior under high concurrency. The mark_evictable/mark_non_evictable calls rely on correct ref_cnt tracking; any mismatch could cause stale entries in evictable_blocks or missing entries. The protected set in evict still works, but the new eviction loop assumes all evictable_blocks have ref_cnt == 0 (assertion added).

**Key insights**  
The main bottleneck was scanning all blocks to find evictable ones. Maintaining a separate evictable list is a classic LRU optimization. The performance gain is substantial (10x throughput) but further improvements are possible in lookup/get. The change is backward-compatible for the public API but alters internal LRU ordering semantics.

### Summary

**What changed and why**  
Added core primitives for partial prefix cache: prefix-chain block hashes at hash_block_size, coarser cache-group block views via BlockHashListWithBlockSize, partial entry registration in BlockPool, and cleanup/promotion semantics. This is the first step toward fine-grained prefix cache hits for hybrid models where full-attention cache block size must align with Mamba state block size.

**Technical impact**  
Introduces new methods cache_partial_block, _remove_cached_block_hashes, _insert_block_hash, and modifies cache_full_blocks to handle partial-to-full promotion. **Potential risks**

The new partial entry logic may introduce bugs in edge cases like concurrent partial registrations or eviction. The removal of the assertion that new full blocks have no hash could mask errors if promotion logic is incorrect.

**Key insights**  
The design keeps full-block and partial-boundary matching on the same hash chain by using chained boundary hashes, avoiding a second hash calculation path. Partial entries are lookup metadata owned by the cache block, with cleanup on eviction/reset/promotion. This PR deliberately stops at primitives; later PRs will add scheduler splitting and backend checkpointing.
