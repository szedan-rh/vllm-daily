  - `vllm/v1/kv_offload/tiering/base.py`
  - `vllm/v1/kv_offload/tiering/manager.py`

### Summary

**What changed and why**  
Replaced the blocking idle-flush in `build_connector_meta` with a non-blocking drain mechanism. Previously, when all requests finished, the scheduler would block on `worker.wait()` to flush in-flight offloads, which could stall new requests. Now, the engine keeps stepping via `has_pending_push_work()` until offloads complete, avoiding blocking. Added `has_pending_work` methods to the base manager and tiering classes to support this. Updated tests to remove expected flush assertions that no longer apply.

**Technical impact**  
Eliminates a blocking call on the engine-core thread, improving responsiveness under offloading workloads. The engine now continues to call `build_connector_meta` and `update_connector_output` even when no requests are scheduled, as long as there are pending offload transfers. This ensures multi-tier offloads (e.g., CPU→storage) drain correctly without holding up new requests. Test coverage is adjusted to reflect the new non-blocking behavior.

**Potential risks**  
The non-blocking drain relies on the engine continuing to step, which may increase CPU usage slightly during idle periods with pending offloads. If `has_pending_push_work` returns True indefinitely due to a bug, the engine could loop forever. The change also removes the explicit flush on idle, so if any code path depends on immediate flush completion, it may break. However, the PR description indicates this is intentional and tested.

**Key insights**  
The core insight is that blocking the engine to flush offloads is unnecessary; instead, the engine can keep stepping to drain pending transfers asynchronously. This is a pattern of replacing synchronous waits with polling via a 'has work' predicate. The change also simplifies the scheduler by removing the special-case flush logic, making the code more uniform.
