### Base Information

- **Author:** [bigPYJ1151](https://github.com/bigPYJ1151)

  - `csrc/cpu/cpu_attn_impl.hpp`
  - `csrc/cpu/cpu_fused_moe.cpp`
  - `csrc/cpu/torch_bindings.cpp`
  - `tests/kernels/attention/test_cpu_attn.py`
  - `vllm/_custom_ops.py`
  - `vllm/v1/attention/backends/cpu_attn.py`

### Summary

**What changed and why**  
Added dynamic causal mask support in CPU attention by introducing an optional `dynamic_causal` tensor that allows per-request causal behavior, replacing the previous global `causal` flag. Also optimized the fused MoE `gelu_tanh` activation by using Sleef vectorized tanh instead of scalar std::tanh. These changes enable the Gemma Diffusion model which requires per-request causal control and accurate tanh computation.

**Technical impact**  
The attention scheduler and main loop now accept a `dynamic_causal` pointer; if non-null, each request's causal flag is read from the tensor, otherwise falls back to the global `causal` flag. The sliding window calculation was refactored into a helper function that takes both window size and per-request causal flag. The Python bindings and test infrastructure were updated accordingly. The MoE change replaces a scalar loop with a vectorized Sleef tanh call, improving performance while maintaining correctness for diffusion models.

**Potential risks**  
The dynamic causal path is new and may have edge cases (e.g., mixing dynamic and static causal requests). The Sleef tanh function may have different numerical behavior than std::tanh, potentially affecting model convergence; the PR notes that fast_exp forms are avoided for correctness. The refactoring of sliding window parameters (combining left/right into a single size) could introduce bugs if callers are not updated consistently. The test coverage for dynamic causal is limited to a single test with alternating boolean pattern.

**Key insights**  
The PR demonstrates a clean extension of the CPU attention kernel to support per-request causal masks without breaking existing functionality. The use of Sleef for vectorized tanh is a good optimization for fused MoE. The refactoring of sliding window parameters simplifies the interface but requires careful validation across all call sites.

### Summary

**What changed and why**

The fix separates request bookkeeping cleanup from block pool return, using a fence mechanism with step sequence numbers and a deferred free queue.

**Technical impact**  
The change introduces a deferred block freeing mechanism in the scheduler, controlled by a new `defer_block_free` flag enabled only when both async scheduling and KV consumer mode are active. It adds `pop_blocks_for_free()` methods across the KV cache manager stack to allow immediate bookkeeping removal without returning blocks to the pool. The scheduler maintains a FIFO queue of deferred frees keyed by a step sequence number, and drains them in `update_from_output()` after the corresponding step's output is processed. This ensures that in-flight GPU writes complete before blocks are reused, preventing data corruption. The change also adds a `last_sched_seq` field to requests and updates the scheduler's `_free_request_blocks` helper to decide between immediate and deferred freeing.

**Potential risks**  
The deferred freeing mechanism adds complexity and could introduce memory pressure if deferred frees accumulate due to bugs in fence tracking or draining. The change modifies core scheduler and KV cache manager code, which could affect performance or correctness in non-deferred paths. The new `pop_blocks_for_free` methods must be correctly implemented in all subclasses (e.g., HybridKVCacheCoordinator, SingleTypeKVCacheManager) to avoid leaks or double frees. The test coverage is extensive but may not cover all edge cases, such as concurrent preemptions or aborts with multiple in-flight steps. The change also removes the `--no-async-scheduling` workaround from some test configurations, which could expose other issues.

**Key insights**  
The race is particularly severe for hybrid/mamba-style caches where in-flight steps rewrite entire state blocks every step, but it also affects full-attention KV blocks. The fix is scoped to the specific condition (async scheduling + KV consumer) to minimize risk. The implementation cleanly separates bookkeeping from block pool return, allowing existing immediate-free behavior to remain unchanged when deferral is disabled. The fence mechanism using step sequence numbers is a robust way to ensure ordering without introducing additional synchronization primitives.
