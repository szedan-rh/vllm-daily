## 7. [[Bugfix][KV Connector] Fix SimpleCPUOffloadScheduler TOCTOU between Phase A and Phase B](https://github.com/vllm-project/vllm/pull/42289)


### Base Information

- **Author:** [qyYue1389](https://github.com/qyYue1389)

- **Type:** `SUBSTANTIAL`
- **[Changed Files](https://github.com/vllm-project/vllm/pull/42289/files) (2):**
  - `tests/v1/simple_kv_offload/test_scheduler.py`
  - `vllm/v1/simple_kv_offload/manager.py`

### Summary

**What changed and why**  
Fixed a TOCTOU race in SimpleCPUOffloadScheduler where CPU cache blocks found in Phase A (get_num_new_matched_tokens) could be evicted before Phase B (update_state_after_alloc) consumed them, causing an assertion crash. The fix pins found blocks immediately in Phase A via cpu_block_pool.touch(), caches the hit info, and consumes it in Phase B without a second search. Added defensive cleanup in request_finished and stale-pin release on retry. Also added a regression test that reproduces the race.

**Technical impact**  
Eliminates a crash (AssertionError) when CPU cache is full and LRU eviction occurs between the two scheduler phases. The change is localized to the scheduler's manager.py and adds a new test. Performance impact is minimal: an extra touch/free pair per CPU hit, which is O(1) per block.

**Potential risks**  
The fix introduces new state (_pending_cpu_hits) that must be correctly cleaned up on all paths (normal, cancellation, preemption, retry). The warning logs for unexpected states are defensive but could mask bugs if the state machine changes. The partial-take slicing assumes num_external_tokens is block-aligned, which is already enforced elsewhere. The stale-pin release on retry may cause a double-free if the pin was already released, but the code pops before releasing, so it should be safe.

**Key insights**  
The root cause is a classic TOCTOU race between two scheduler phases that both searched the CPU cache independently. Pinning the blocks in Phase A and caching the result avoids the second search entirely. The fix also handles edge cases like partial token budget and request cancellation, showing careful state management. The regression test is well-designed to force LRU eviction between phases.

### Summary

**What changed and why**  
Fixed off-by-one grid dimensions in three CUDA kernels (moe_lora_align_block_size_kernel, lora_count_and_sort_expert_tokens_kernel, moe_lora_align_block_size_small_batch_expert_kernel) in csrc/moe/moe_align_sum_kernels.cu. The grid was sized to max_loras but active_lora_ids can contain max_loras+1 entries (including -1 for base-model tokens), causing the real LoRA slot to be skipped and output buffers left uninitialized, leading to illegal memory access. Added defensive lora_id >= max_loras guard to all three kernels and added a regression test.

**Technical impact**  
Fixes CUDA illegal memory access in LoRA MoE pipeline when max_loras equals the number of real active LoRAs and the batch contains base-model tokens. The fix ensures the align kernel processes all LoRA slots correctly, preventing downstream kernel failures. The new test verifies the fix and prevents regression.

**Potential risks**  
Low risk. The change is minimal (grid size +1 and an extra guard condition) and mirrors an existing fix in the Triton kernel. The guard condition is short-circuit evaluated to avoid out-of-bounds access. The test covers the mixed base+LoRA case, disabled adapters, and out-of-range lora_id. No performance impact expected as the extra grid block is trivial.

**Key insights**  
The root cause was a subtle off-by-one error: active_lora_ids has length max_loras+1 to accommodate the -1 sentinel for base-model tokens, but the grid was sized to max_loras, causing the real LoRA slot to be skipped when -1 occupies position 0. The fix aligns the C++ kernel grid with the Triton kernel fix from PR #32277. The defensive guard provides an additional safety net against unexpected lora_id values.
