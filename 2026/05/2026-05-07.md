  - `vllm/model_executor/layers/mamba/gdn_linear_attn.py`

### Summary

**What changed and why**  
This PR fuses multiple Triton kernels in the GatedDeltaNetAttention decode path to reduce launch overhead. It integrates AITER's optimized kernels for convolution update, gating, and delta rule, and restructures the forward pass to separate prefill/speculative-decode from the decode-only hot path. The changes improve throughput by ~5.7% and reduce TPOT by ~5.9% on AMD MI355x.

**Technical impact**  
The PR adds a new decode fast-path method (_forward_core_decode_fast) that uses fused AITER Triton kernels, reducing kernel launch count per layer. It also refactors the forward method to use a compiled helper (prepare_gdn_attention_core_inputs) for prefill, and passes core_attn_out through chunk kernels to eliminate a copy. The changes are gated behind a runtime check (are_gdn_triton_kernels_available()) for graceful fallback.

**Potential risks**  
The fast-path is only available on ROCm with AITER kernels; other backends are unaffected. The fused kernels may have edge cases not covered by tests (e.g., different sequence lengths, head dimensions). The refactored forward method changes control flow, which could introduce bugs in prefill or speculative-decode paths. The core_attn_out buffer reuse assumes inference-only and matching dtypes, which could cause silent corruption if misused.

**Key insights**  
The PR demonstrates significant performance gains (5-8%) from kernel fusion in attention layers, highlighting the importance of reducing launch overhead in transformer inference. The use of torch.compile with fused cat operations to minimize copy kernels is a notable optimization technique. The graceful fallback mechanism ensures backward compatibility with older AITER builds.

  - `vllm/v1/worker/gpu/buffer_utils.py`
  - `vllm/v1/worker/gpu/sample/penalties.py`

### Summary

**What changed and why**  
This PR eliminates unnecessary GPU<->CPU synchronizations in multiple attention backends (flashinfer, flex_attention, mamba_attn, tree_attn, triton_attn, turboquant_attn) and utility functions. Changes include: replacing scalar tensor assignments with slice assignments to avoid syncs, using CPU-resident copies of metadata (query_start_loc_cpu, seq_lens_cpu_upper_bound) to avoid D2H transfers, replacing torch.bincount with scatter_add_ to avoid sync, and using async_tensor_h2d for non-blocking host-to-device transfers. These changes were motivated by profiling that identified syncs as performance bottlenecks.

**Technical impact**  
The changes improve throughput by ~4.8% in TurboQuant benchmarks (e.g., output throughput from 7441 to 7802 tok/s) and reduce latency (e.g., mean TPOT -4.66%, mean ITL -4.62%). The modifications are backward-compatible: they preserve existing behavior while making operations asynchronous where possible. The addition of CPU-resident metadata fields (query_start_loc_cpu, seq_lens_cpu) and precomputed max lengths (max_query_len_prefill, etc.) avoids per-forward syncs in properties like prefill_metadata and decode_metadata.

**Potential risks**  
The use of seq_lens_cpu_upper_bound (which may be optimistic for decode rows) could lead to over-allocation in workspace sizing, but this is safe and already handled. The removal of pin_memory=True in some async_tensor_h2d calls (buffer_utils.py, penalties.py) relies on the new default PIN_MEMORY logic, which disables pinning on WSL; this could cause performance regression on WSL if pinning was beneficial. The scatter_add_ replacement for bincount assumes request_ids are contiguous and within bounds; if not, it could produce incorrect results. The flex_attention changes to unique_static_unsorted use a different packing strategy that may alter behavior for edge cases (e.g., all elements kept).

**Key insights**  
The PR demonstrates that many GPU-CPU syncs in attention backends are unnecessary and can be avoided by careful use of CPU-resident data, non-blocking transfers, and alternative tensor operations. The performance gains (4-5% throughput) are significant for a relatively small code change. The approach of precomputing metadata on CPU and passing it alongside GPU tensors is a reusable pattern for other backends. The use of slice assignments (e.g., tensor[:1] = value) instead of scalar assignment (tensor[0] = value) is a subtle but effective technique to avoid syncs.
