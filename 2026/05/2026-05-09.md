### Summary

**What changed and why**  
Added a notification path so that when a KV-transfer request is rejected on the decode node before engine admission, the prefill node is immediately informed to free stranded KV blocks. Previously, blocks would remain pinned until the abort timeout (default 480s). The change introduces a new method `notify_kv_transfer_request_rejected` in the engine client protocol, wraps the OpenAI serving layer's create methods to detect pre-admission rejections, and creates a synthetic 'abort immediately' request in the engine core to trigger the connector's cleanup hook.

**Technical impact**  
Adds a new abstract method to `EngineClientProtocol` and implements it across all engine client variants (in-process, sync MP, async MP, DPLB async MP). Modifies the scheduler to report pending connector metadata so that the cleanup tick occurs even when there is no other work. Changes the `_reqs_need_recv` value type from `(Request, BlockIds)` to `(dict[str, Any], BlockIds)` to accommodate pre-admission rejections that lack a full Request object. The serving layer now wraps each create coroutine with a cleanup wrapper that checks for `do_remote_prefill` and calls the new notification on error or error response.

**Potential risks**  
The new notification path adds overhead to every pre-admission rejection, but only when `do_remote_prefill` is set, which is a small fraction of requests. The synthetic 'abort immediately' request uses a dummy prompt token and sampling params, which could interact with other scheduler logic (e.g., token counting, metrics). The `has_kv_connector` check in the serving layer uses `vllm_config` which may not be present in all configurations; the code handles this with `getattr` defaults. There is a risk of double notification if the request is both rejected pre-admission and later aborted normally, but the `do_remote_prefill` flag is consumed to prevent re-issuing.

**Key insights**  
The fix leverages the existing `request_finished` hook in the NIXL connector by creating a synthetic request that is immediately aborted, thus reusing the cleanup logic without duplicating it. The approach is generic enough to work with any connector that implements `request_rejected_before_admission`. The serving layer wrapper pattern (`_with_kv_transfer_rejection_cleanup`) cleanly separates concerns without modifying each rejection path individually. The change also adds unit tests for the connector behavior and the multi-connector fan-out.

## 5. [[Bugfix] Fix DeepSeek v4 topk numerical issue for unaligned max-model-len](https://github.com/vllm-project/vllm/pull/42169)


### Base Information

- **PR Number:** #42169
- **Author:** [wzhao18](https://github.com/wzhao18)

- **Merged time:** 2026-05-09 20:30:09
- **Type:** `SUBSTANTIAL`
- **[Changed Files](https://github.com/vllm-project/vllm/pull/42169/files) (1):**
  - `csrc/topk.cu`

### Summary

**What changed and why**  
Changed stride calculation from logits.size(1) to logits.stride(0) in two places in csrc/topk.cu. The previous code assumed contiguous logits, but when max_model_len is not aligned to 256, the logits tensor becomes non-contiguous due to padding in deepgemm's fp8_fp4_paged_mqa_logits, causing incorrect stride and numerical issues. Using stride(0) correctly accounts for the memory layout.

**Technical impact**  
Fixes a numerical bug that caused distribution shift in generated tokens and degraded accuracy (e.g., GPQA pass@1 dropped from ~89% to ~82% with max_model_len=900000). After fix, accuracy and token distribution match the baseline without max_model_len. The change is minimal (2 lines) but critical for correctness when using unaligned max_model_len.

**Potential risks**  
Low risk. The change only affects the stride calculation, which is now correct for both contiguous and non-contiguous tensors. Existing contiguous cases remain unaffected. No new dependencies or performance regressions expected.

**Key insights**  
The bug was subtle: using size(1) assumes contiguous layout, but deepgemm's padding creates non-contiguous tensors. Using stride(0) is the correct way to get the distance between rows. This highlights the importance of handling tensor memory layout correctly in CUDA kernels, especially when interfacing with external libraries that may modify tensor strides.

**Key insights**  
The change leverages a CUDA driver feature to optimize memory transfer bandwidth by relaxing ordering constraints, demonstrating a nuanced understanding of the synchronization guarantees in the offloading pipeline. ### Summary

**What changed and why**  
Fixed two bugs in Sequence Parallelism (SP) for multimodal models and text models with --enable-prompt-embeds. The test coverage is limited to small models and specific parallel setups.

**Key insights**  
The pattern matcher replaces from the end of the graph, so temporary slices inserted by earlier replacements become no-ops after later replacements shrink the residual. The fix uses tp_rank-aware slicing to handle cases where the first RMSNorm pattern is never matched (e.g., when inputs_embeds are passed directly). The EP+LoRA integration is new and may have edge cases, especially with different all2all backends or sequence parallelism (explicitly not supported). The slicing of LoRA weights in the model manager assumes a specific weight layout; any change in the checkpoint format could break this. The removal of LoRA restrictions for int8/mxfp4/mxfp8 backends may expose untested code paths. The changes to the punica wrapper's moe_lora_align_block_size to handle global expert indices could affect non-EP paths if not carefully guarded.

**Key insights**  
The PR carefully handles the interaction between EP and fully_sharded LoRA, asserting they are mutually exclusive. It introduces a local_token_lora_mapping field in MoELoRAContext to pass the per-rank mapping after EP dispatch. The EP slicing of LoRA weights is done once at add time to avoid re-slicing on every activation. The changes to the oracle files indicate a broader effort to unify LoRA support across quantization backends.
