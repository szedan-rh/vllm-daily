## 1. [Reduce memory usage for granite_speech.](https://github.com/vllm-project/vllm/pull/42933)


### Base Information

- **PR Number:** #42933
- **Author:** [Yihuki](https://github.com/Yihuki)
- **Merged By:** [DarkLight1337](https://github.com/DarkLight1337)
- **Merged time:** 2026-05-24 23:12:58
- **Type:** `SUBSTANTIAL`
- **[Changed Files](https://github.com/vllm-project/vllm/pull/42933/files) (1):**
  - `vllm/model_executor/models/granite_speech.py`

### Summary

**What changed and why**  
Replaced torch.sum with torch.einsum in the relative positional embedding computation to reduce memory usage. The original implementation created a large intermediate tensor by expanding rel_pos_emb and multiplying with query_states, consuming over 10GB VRAM for the granite-speech-4.1-2b model. The new einsum approach computes the same result without materializing the full intermediate matrix.

**Technical impact**  
Reduces peak memory usage during forward pass for granite_speech models, particularly for the 2B parameter variant. The change is mathematically equivalent but more memory-efficient. No change in model output or accuracy.

**Potential risks**  
Low risk. The einsum operation is well-defined and the dimensions match. However, if the model is used with different input shapes or on hardware with limited einsum support, there could be issues. The change is localized to one line and covered by existing tests.

**Key insights**  
The PR demonstrates a common optimization pattern: replacing explicit broadcasting and summation with einsum to avoid large intermediate tensors. This is especially important for large models where memory is a bottleneck. The fix is minimal (1 line changed) but has significant impact on memory usage.

Added support for DSV4 (different block sizes for scheduler and hash) in SimpleCPUOffloadBackend. **Technical impact**  
The scheduler now uses separate block sizes for scheduling and hashing, and the code correctly handles cases where block sizes differ across KV cache groups. The modification to skip blocks without hashes in eager store may change behavior for non-DSV4 models, but should be safe as those blocks would never be cache hits. The test suite may need expansion to cover DSV4 scenarios.

**Key insights**  
The PR demonstrates how to handle heterogeneous block sizes in a KV cache system, which is a common requirement for advanced attention mechanisms like DSV4. The fix to skip SWA blocks instead of breaking is a subtle but important correctness improvement for prefix caching.

The alignment-based skip optimization reduces SWA store operations by ~78% for 100K token sequences. The hash block size fix ensures correct prefix caching behavior. The empty shared_by filter prevents crashes when reserved memory slots have no corresponding model layers.

**Potential risks**  
The alignment optimization assumes a single consistent full-attention block size; if multiple full-attention groups with different sizes exist, the optimization is disabled. The skip logic relies on correct computation of `alignment_block_count` and `sliding_window_size_in_blocks`; errors could lead to missing stores and incorrect loads. The change to use `resolve_kv_cache_block_sizes` may alter hash block size for non-DSv4 models, but the assertion ensures divisibility. The empty shared_by filter may hide bugs if tensors are unintentionally skipped.

**Key insights**  
The alignment-based skip optimization is a clever way to reduce offloading overhead for hybrid architectures by leveraging the fact that load hits are aligned to full-attention block boundaries. The fix for empty shared_by highlights a design pattern where reserved memory slots are created but not used. The use of `resolve_kv_cache_block_sizes` centralizes block size resolution, improving maintainability.
