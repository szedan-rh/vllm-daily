### Summary

**What changed and why**  
Fixed a bug where LoRA kernels were not captured in cudagraphs during speculative decoding, causing silent LoRA skipping. The root cause was that the dummy LoRAMapping(0,0) workaround in V2's _run_model_forward set no_lora_flag_cpu=True, which short-circuited the Python wrapper and prevented LoRA kernel launches at capture time. The fix adds a new method prepare_for_cudagraph_capture() that sets no_lora_flag_cpu=False and num_active_loras_cpu=1, ensuring the kernels are captured but early-exit with no work. **Technical impact**  
Fixes a correctness bug in speculative decoding with LoRA adapters, restoring match ratio from 76/100 to 100/100. The change ensures that LoRA kernels are properly captured in cudagraphs during the dummy run, so they execute correctly during real inference. The test improvements (try/finally, better assertion messages) make the test more robust and debuggable.

**Potential risks**  
Low risk. The change only affects the dummy run path for cudagraph capture, and the kernel early-exit logic remains unchanged. The new code iterates over punica_wrapper_mapping values, which could be a performance concern if there are many wrappers, but the set deduplication mitigates repeated work. The test changes are purely structural and do not alter test logic.

**Key insights**  
The bug was subtle: the dummy LoRAMapping caused the Python wrapper to skip kernel launches entirely, so the cudagraph captured nothing. The fix ensures the kernel is launched but does no work via early-exit, preserving the cudagraph structure. This pattern (capture with no-op) is a useful technique for handling conditional execution in cudagraphs.

- **Type:** `SUBSTANTIAL`
- **[Changed Files](https://github.com/vllm-project/vllm/pull/43020/files) (1):**
  - `vllm/device_allocator/cumem.py`

### Summary

**What changed and why**  
Added a `torch.cuda.synchronize` call in `_python_free_callback` before the C extension unmaps and releases memory. This fixes a race condition where in-flight CUDA kernels could still be accessing memory that was freed, causing illegal memory access errors.

**Technical impact**  
Ensures that all pending CUDA operations on the stream associated with the allocation handle complete before the memory is unmapped and released. This prevents CUDA_ERROR_ILLEGAL_ADDRESS errors that occurred with sleep-mode allocator when pool-backed tensors were freed while kernels were still running.

**Potential risks**  
The synchronize call adds a small performance overhead because it blocks the CPU until all pending GPU work on that stream completes. However, this is necessary for correctness and the overhead is expected to be minimal in practice since the free callback is not on a hot path.

**Key insights**  
The bug was caused by the pluggable allocator not deferring memory reclaim like the regular caching allocator does. The fix uses stream synchronization to ensure memory is not released while still in use, which is a simple and effective solution for this race condition.

## 15. [[BugFix] Use correct logprobs for `logprob_token_ids`](https://github.com/vllm-project/vllm/pull/43125)


### Base Information

- **PR Number:** #43125
- **Author:** [njhill](https://github.com/njhill)

- **Merged time:** 2026-05-21 15:42:21
- **Type:** `SUBSTANTIAL`
- **[Changed Files](https://github.com/vllm-project/vllm/pull/43125/files) (1):**
  - `vllm/v1/sample/sampler.py`

### Summary

**What changed and why**  
The fix ensures that when gathering logprobs for specific token IDs (logprob_token_ids), the method uses the already computed raw_logprobs instead of recomputing logprobs from logits via a Triton kernel. This honors the logprobs mode (raw_logprobs vs raw_logits) and avoids unnecessary recomputation. The rank computation is also updated to use logprobs instead of logits, keeping it consistent with the top-k logprobs case. Additionally, the condition for computing raw_logprobs is extended to include cases where logprob_token_ids is present, and the greedy sample path similarly checks for logprob_token_ids.

**Technical impact**  
Improves performance by avoiding redundant logprobs computation when logprob_token_ids is requested. Ensures correctness by using the same logprobs representation as the top-k case, which is important for consistency in generative scoring. The change also replaces a Triton kernel call with a simple gather operation, which may affect performance characteristics but is more straightforward and avoids potential mode mismatches.

**Potential risks**  
The change removes the use of the fused Triton kernel (compute_token_logprobs) which was benchmarked as ~1.4x faster for batch sizes >1. The new gather-based approach may be slower for large batches. The rank computation now uses batched_count_greater_than on logprobs instead of logits, which should be equivalent due to monotonicity of log_softmax, but any numerical differences could affect rank results. The addition of torch._dynamo.decorators.mark_unbacked may affect compilation behavior.

**Key insights**  
The bug was that logprob_token_ids always recomputed logprobs from logits regardless of the logprobs mode, leading to inconsistency and extra computation. The fix aligns the behavior with the top-k logprobs path. The change is specific to MRV1; MRV2 does not have this issue. The update to the greedy sample path ensures logprobs are computed when logprob_token_ids is present, which was previously missing.
