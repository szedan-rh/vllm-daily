## 2. [[Bugfix][Frontend] streaming tool-call serializer drops first args chunk when name and args share a DeltaMessage](https://github.com/vllm-project/vllm/pull/42683)


### Base Information

- **PR Number:** #42683
- **Author:** [ignaciosica](https://github.com/ignaciosica)
- **Merged By:** [sfeng33](https://github.com/sfeng33)
- **Merged time:** 2026-05-27 22:20:56
- **Type:** `SUBSTANTIAL`
- **[Changed Files](https://github.com/vllm-project/vllm/pull/42683/files) (4):**
  - `tests/entrypoints/openai/responses/test_serving_responses.py`
  - `tests/entrypoints/openai/responses/test_streaming_events.py`
  - `vllm/entrypoints/openai/responses/serving.py`
  - `vllm/entrypoints/openai/responses/streaming_events.py`

### Summary

**What changed and why**  
Fixed a bug in the streaming tool-call serializer where the first arguments chunk was dropped when a DeltaMessage contained both a name and arguments DeltaToolCall at the same index. Added a `split_delta` function to decompose compound DeltaMessages into atomic deltas before state machine processing, and updated the streaming loop in serving.py and the emit_delta method to use it. Added comprehensive unit tests for split_delta and the processor, plus an integration test for the compound content and tool name/args scenario.

**Technical impact**  
Ensures that tool parsers like KimiK2ToolParser, which emit name and arguments in separate DeltaToolCalls within a single DeltaMessage, no longer lose the first arguments chunk. This fixes non-deterministic JSON parsing failures in streaming tool calls, affecting ~60% of tool calls in some deployments. The change introduces a new `split_delta` utility and modifies the state machine to handle multiple atomic deltas per original message, improving correctness without breaking existing behavior.

**Potential risks**  
The new `split_delta` function changes the order of event emission: reasoning, then content, then tool calls grouped by index. This could affect downstream consumers that rely on the previous ordering (though the old behavior was buggy). The change also modifies `emit_delta` to concatenate arguments from multiple tool calls at the same index, which could alter behavior if multiple arguments chunks are expected separately. However, the tests cover these scenarios and the logic is consistent with the API spec.

**Key insights**  
The root cause was that `emit_delta` only inspected `delta_message.tool_calls[0]`, ignoring subsequent tool calls at the same index. The fix generalizes the state machine to handle compound deltas by splitting them into atomic units, which also simplifies the special case for reasoning-to-content transitions. The new `split_delta` function is a clean abstraction that improves maintainability and testability.
