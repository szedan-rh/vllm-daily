## 1. [[Bugfix] Fix RMSNorm kernels to multiply in weight's native dtype](https://github.com/vllm-project/vllm/pull/42379)


### Base Information

- **PR Number:** #42379
- **Author:** [liulanze](https://github.com/liulanze)
- **Merged By:** [vllm-bot](https://github.com/vllm-bot)
- **Merged time:** 2026-05-29 23:16:53
- **Type:** `SUBSTANTIAL`
- **[Changed Files](https://github.com/vllm-project/vllm/pull/42379/files) (2):**
  - `csrc/libtorch_stable/layernorm_kernels.cu`
  - `csrc/libtorch_stable/layernorm_quant_kernels.cu`

### Summary

**What changed and why**  
Fixed a regression in RMSNorm CUDA kernels where the weight was upcast to float32 before multiplication, causing precision loss. The fix restores multiplication in the weight's native dtype (BF16/FP16) to match the Python spec and v0.19.1 behavior. Changes were applied to both layernorm_kernels.cu and layernorm_quant_kernels.cu across vectorized, optimized, and scalar paths.

**Technical impact**  
Eliminates ~0.03 per-element error per layer that compounded catastrophically in models with many RMSNorm layers (e.g., LLaDA-2, DeepSeek). Restores bit-exact agreement with the unfused composite path and passes all 2307 tests. The fix is minimal (10 lines changed) and does not affect performance since the multiplication is still done in the weight's native type.

**Potential risks**  
Low risk. The change is a straightforward revert of a previous regression and has been verified locally on A100 with comprehensive test suites. The only risk is if other code paths depend on the FP32 weight multiplication behavior, but the PR description indicates the Python spec mandates native dtype multiplication.

**Key insights**  
The bug was introduced in a previous PR (#40860) that added DeepSeek V4 support but inadvertently changed the multiplication order. The fix highlights the importance of maintaining dtype consistency in fused kernels to avoid silent accuracy degradation. The quant kernels required the same fix to stay consistent with the non-fused path, demonstrating the need for cross-kernel alignment.

## 6. [offload prompt_embeds decode in render_prompts_async to avoid blocking](https://github.com/vllm-project/vllm/pull/43792)


### Base Information

- **PR Number:** #43792
- **Author:** [gagandhakrey](https://github.com/gagandhakrey)
- **Merged By:** [DarkLight1337](https://github.com/DarkLight1337)
- **Merged time:** 2026-05-29 18:36:35
- **Type:** `SUBSTANTIAL`
- **[Changed Files](https://github.com/vllm-project/vllm/pull/43792/files) (1):**
  - `vllm/renderers/base.py`

### Summary

**What changed and why**  
Changed render_prompts_async from a synchronous wrapper to an actual async implementation that offloads prompt_embeds decoding to a thread pool executor. Previously, it called the synchronous render_prompts, which blocked the event loop during base64 decode, torch.load, and tensor operations. Now it uses asyncio.gather with a new _render_prompt_async helper that awaits safe_load_prompt_embeds_async, bound to self._executor.

**Technical impact**  
Eliminates event loop blocking on prompt_embeds decode, reducing tail latency for concurrent requests. Multiple prompt_embeds in a single request are now decoded in parallel. The change is scoped to the completions endpoint path; chat endpoint was already async. The executor size is tunable via renderer_num_workers.

**Potential risks**  
If the executor is shared with other async operations, contention could arise under high load. The empty-list validation was moved from render_prompts to render_prompts_async, which changes error behavior slightly (now raises ValueError instead of returning empty list). The new async path may introduce overhead for non-bytes prompts due to asyncio.gather. Thread safety of safe_load_prompt_embeds should be verified.

**Key insights**  
The fix addresses a real performance issue where synchronous decode blocked the event loop for tens to hundreds of milliseconds. The approach mirrors existing patterns in the codebase (e.g., _clear_mm_cache_async). The change is minimal (+21/-1 lines) and focused, with clear test plan and no behavioral changes for non-bytes prompts.

  - `vllm/model_executor/models/minimax_m2.py`

### Summary

**What changed and why**  
Added a fused FP32 router GEMM kernel for MiniMax-M2 MoE gate, which previously required separate BF16-to-FP32 conversion and split-K GEMM kernels. The new kernel fuses these operations into a single kernel, reducing kernel launch overhead and improving performance, especially at low concurrency. The implementation includes a new CUDA kernel, Python bindings, benchmark script, tests, and integration into the GateLinear layer and MiniMax-M2 model.

**Technical impact**  
Introduces a new CUDA kernel (fp32_router_gemm) that supports BF16 or FP32 activation input and FP32 weight, outputting FP32. The kernel is specialized for H=3072, E=256, M<=32 and requires SM90+ and CUDA >=12.0. It is integrated into the GateLinear layer as Tier 2 dispatch, with a custom op for torch.compile compatibility. The CMake build system is updated to conditionally compile the kernel. Performance improvements of up to 32% in output token throughput and 29% reduction in TTFT at low concurrency are observed, with no accuracy regression.

**Potential risks**  
The kernel is only built for SM90+ and CUDA >=12.0, so older hardware will fall back to existing paths. The kernel is specialized for specific dimensions (H=3072, E=256, M<=32), so it will not be used for other configurations. The TTFT regression at concurrency=16 (+10%) is noted and may need investigation. The custom op registration for torch.compile may have compatibility issues with future PyTorch versions. The kernel uses programmatic stream serialization, which may interact with other CUDA operations.

**Key insights**  
The fused kernel eliminates the need for a separate dtype conversion kernel and reduces the number of kernel launches from up to 3 to 1, significantly improving performance at low concurrency. The implementation is adapted from the existing dsv3_router_gemm kernel, demonstrating code reuse. The benchmark shows consistent improvements across low-to-mid concurrency, with the largest gains at concurrency=2. The accuracy comparison shows no regression, with slight improvements on GPQA Diamond. The kernel is conditionally compiled and only enabled for supported architectures, ensuring backward compatibility.
