## 2. [[CPU] Enable non-divisible GQA for decode workitems in mixed batches](https://github.com/vllm-project/vllm/pull/43032)


### Base Information

- **PR Number:** #43032
- **Author:** [zhejiangxiaomai](https://github.com/zhejiangxiaomai)
- **Merged By:** [bigPYJ1151](https://github.com/bigPYJ1151)
- **Merged time:** 2026-05-25 23:15:48
- **Type:** `SUBSTANTIAL`
- **[Changed Files](https://github.com/vllm-project/vllm/pull/43032/files) (1):**
  - `csrc/cpu/cpu_attn_impl.hpp`

### Summary

**What changed and why**  
Relaxed the condition for enabling GQA on CPU decode workitems from requiring exact divisibility (max_num_q_per_iter % q_heads_per_kv == 0) to allowing any ratio where q_heads_per_kv <= max_num_q_per_iter. For mixed batches, GQA is now applied per-request only for single-token decode workitems, while prefill remains on MHA. The scheduler and main loop logic were updated to compute per-workitem GQA flags and adjust scratchpad sizing accordingly.

**Technical impact**  
Enables GQA for non-divisible q/kv head ratios on decode workitems, improving performance for models like LLaMA-3.1-70B (8 KV heads, 64 Q heads) where max_num_q_per_iter may be 32. Mixed batches benefit from per-request GQA without affecting prefill. Scratchpad memory allocation is reduced when GQA is used (num_heads_kv instead of num_heads_q).

**Potential risks**  
The per-request GQA decision adds a loop over workitem groups in the main loop, which may introduce slight overhead. The new logic assumes that decode workitems have q_token_num == 1; if this assumption is violated, behavior may be incorrect. The change also modifies scratchpad sizing, which could cause out-of-memory if the new size is underestimated, though the code uses the correct formula.

**Key insights**  
The key insight is that for decode (single token), GQA can be applied as long as q_heads_per_kv <= max_num_q_per_iter, because each workitem processes at most one token. The previous divisibility requirement was overly restrictive. The PR correctly separates the GQA decision for scratchpad allocation (global) from the per-workitem GQA usage, allowing mixed batches to use GQA only for decode workitems.

It also refactors the backend selection logic, adds a new cute_utils module with shared utilities, updates the engine args, and adds correctness tests.

**Technical impact**  
Introduces a new GDN prefill backend that can be significantly faster than FlashInfer or Triton/FLA under certain workloads, especially with low number of heads (TP) and requests (long prefill with chunking). The PR also modifies existing backend selection logic, which could affect FlashInfer fallback behavior.

**Key insights**  
The CuteDSL kernel outperforms FlashInfer and Triton/FLA in many configurations, particularly with high TP and low V heads, and with skewed sequence lengths. The use of Tensor Cores for matrix inversion is a novel approach that alleviates a known bottleneck. The kernel is JIT-compiled, allowing adaptation to various head configurations. The PR includes extensive microbenchmarks and end-to-end throughput comparisons, showing up to 22% improvement over FLA in some scenarios.

### Summary

**What changed and why**

**Technical impact**  
Prevents silent KV-cache corruption when speculative decoding with Eagle/MTP is used on MooncakeStoreConnector loads. The bug caused uninitialized KV blocks to be read, leading to generation corruption (hallucinated prompts, dropped tokens). Non-MTP workloads are unaffected.

**Potential risks**  
Low risk. The change is scoped to the coordinator's load_mask path and is guarded by a new parameter defaulting to True for existing callers. The regression tests cover the main scenarios (full attention, hybrid, non-eagle).

**Key insights**  
The bug was a subtle double-application of the Eagle pop: lookup applies it to compute hit_length, then load_mask re-applied it, shortening the mask by one block. The fix ensures load_mask uses the already-pruned length without further modification.

### Summary

**What changed and why**  
Fix a crash in MooncakeStoreScheduler when request_finished is called for a request whose tracker was already removed due to preemption. The fix replaces the assert with a graceful check and resets the tracker on preemption instead of removing it, preventing the assertion error that killed the EngineCore process.

**Technical impact**  
Prevents server crashes under high KV-cache pressure when preempted requests are later finished. The change ensures that request_finished handles missing or reset trackers safely, returning (False, None) instead of asserting. The tracker reset on preemption preserves the tracker object but clears its state, allowing subsequent finish calls to be no-ops.

**Potential risks**  
Low risk. The change is defensive and only affects the preemption path. The reset method is new but only called during preemption. Existing behavior for non-preempted requests is unchanged. The tests cover the preemption scenarios.

**Key insights**  
The root cause was a race between preemption and request completion under high load. The fix aligns with the principle of graceful degradation: instead of crashing, the connector safely ignores the redundant finish call. The addition of a reset method on RequestTracker allows reusing the tracker object without stale state.
