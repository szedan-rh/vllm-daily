## 1. [[Bugfix] Skip PP sampled-token receive on last rank during async scheduling](https://github.com/vllm-project/vllm/pull/40749)

- **Author:** [wi-adam](https://github.com/wi-adam)

  - `tests/v1/worker/test_gpu_model_runner.py`
  - `vllm/v1/worker/gpu_model_runner.py`

### Summary

**What changed and why**  
Changed the condition in `sample_tokens()` from checking `world_size > 1` to checking `not is_last_rank` to avoid calling the PP sampled-token receive path on the last pipeline-parallel rank during async scheduling. The last rank broadcasts sampled token IDs, so it should not receive them. This fixes an assertion failure in the empty `execute_model_state` path.

**Technical impact**  
Fixes a bug where the last PP rank would incorrectly attempt to receive sampled token IDs, causing an assertion failure. The fix ensures only non-last PP ranks perform the receive, preserving correct token ID propagation. Two new unit tests verify the rank-selection logic and that PP group lookup is skipped when async scheduling is disabled.

**Potential risks**  
Low risk. The change is minimal (one line) and only affects the condition for calling the receive helper. The new tests cover both the last and non-last rank cases, as well as the async scheduling disabled path. The patch has been verified in a production-like environment with PP=2.

**Key insights**  
The bug was discovered during Gemma 4 31B FP8 inference with PP=2 on AMD RDNA4 GPUs. The fix clarifies the contract: the last PP rank broadcasts, others receive. The existing lazy PP-group lookup is preserved when async scheduling is disabled, avoiding unnecessary overhead.

## 5. [[Build] Fall back to system libgomp when torch has no vendored copy](https://github.com/vllm-project/vllm/pull/40575)


### Base Information

- **PR Number:** #40575
- **Author:** [lyd1992](https://github.com/lyd1992)
- **Merged By:** [bigPYJ1151](https://github.com/bigPYJ1151)
- **Merged time:** 2026-05-05 20:42:03
- **Type:** `SUBSTANTIAL`
- **[Changed Files](https://github.com/vllm-project/vllm/pull/40575/files) (1):**
  - `cmake/cpu_extension.cmake`

### Summary

**What changed and why**  
Modified cmake/cpu_extension.cmake to fall back to system libgomp when PyTorch has no vendored copy. Previously, find_library used PATHS with NO_DEFAULT_PATH and REQUIRED, which failed on platforms like RISC-V, s390x, and Fedora/RHEL aarch64 where PyTorch is built from source or distro-packaged without a vendored libgomp. Now, if the shim dir is empty, it falls back to a standard find_library without NO_DEFAULT_PATH, allowing the system libgomp to be found.

**Technical impact**  
Unblocks CPU extension builds on platforms without vendored libgomp (RISC-V, s390x, Fedora/RHEL aarch64). No behavior change for x86_64 pip-wheel installs where vendored libgomp exists. The LD_LIBRARY_PATH is only set when the vendored copy is actually used.

**Potential risks**  
Low risk. The change is conditional: if shim dir is non-empty, behavior is identical to before; if empty, it falls back to system libgomp, which is the intended fix. Potential risk if system libgomp is incompatible with PyTorch's expectations, but that is unlikely and would have been a pre-existing issue on those platforms.

**Key insights**  
The root cause was diagnosed manually on RISC-V hardware. The fix is minimal and targeted, using a conditional to handle the two cases. The PR description is thorough, including test plan and AI assistance disclosure.
