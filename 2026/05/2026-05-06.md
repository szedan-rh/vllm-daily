- **Type:** `SUBSTANTIAL`
- **[Changed Files](https://github.com/vllm-project/vllm/pull/41665/files) (1):**
  - `csrc/topk.cu`

### Summary

**What changed and why**  
Removed the conditional guard `if (needs_cooperative)` around `cudaMemsetAsync` in `launch_persistent_topk` so that the workspace zeroing is always performed. Previously, the memset was only executed when `needs_cooperative` was true, which is never the case during CUDA graph capture when `max_seq_len` is below the radix threshold. This caused the workspace state to persist across replays, leading to a latent correctness bug if the runtime later takes the radix path.

**Technical impact**  
The memset is now unconditionally issued, ensuring it is captured as a separate node in the CUDA graph. This prevents accumulation of stale workspace state (e.g., arrival_counter) across replays, fixing a potential hang or incorrect behavior when the radix path is used. The performance cost is negligible (sub-microsecond) because the state size is small (~100 KB for largest grids).

**Potential risks**  
Low risk. The change removes a conditional that previously avoided an unnecessary memset for non-cooperative cases. The memset is now always performed, but its cost is minimal. There is no change to kernel logic or data dependencies. The only risk is if the workspace zeroing interferes with some other use of the workspace, but the workspace is dedicated to this kernel and the memset is stream-ordered before the kernel launch.

**Key insights**  
The bug was subtle: the conditional guard was evaluated at capture time, not at replay time, so the memset was never captured when the condition was false during capture. This is a common pitfall with CUDA graph capture—host-side conditionals that depend on runtime state must be avoided or handled carefully. The fix makes the memset unconditional, which is both simpler and correct.

This could incorrectly unwrap if the schema actually expects a parameter named 'arguments' or 'input' with a string value. The change is backward-compatible for existing valid outputs, but may alter behavior for edge cases where models output these patterns incorrectly.

The change modifies core streaming logic in the abstract parser and the chat completion stream generator. There is a risk of regression for other models or configurations that rely on the previous behavior, especially when both reasoning and tool parsers are active. The new tests cover several scenarios, but edge cases with different tokenizers or chat templates may not be fully covered. The fallback condition for content-only delta was changed from checking if both parsers are None to checking if no phase is active, which could affect behavior when parsers are present but not yet in a phase.

**Key insights**  
The bug was caused by inconsistent handling of prompt_token_ids between streaming and non-streaming paths. The fix ensures that the reasoning-end state is determined from the rendered prompt tokens before streaming begins, rather than relying on potentially missing prompt_token_ids in streaming chunks. The PR also improves code clarity by removing redundant checks in _in_reasoning_phase and _in_tool_call_phase, and by adding comprehensive unit tests for the parser's delta parsing logic.

## 29. [[Bugfix] Fix codegen for unqualified names](https://github.com/vllm-project/vllm/pull/40726)


### Base Information

- **PR Number:** #40726
- **Author:** [Lucaskabela](https://github.com/Lucaskabela)
- **Merged By:** [vllm-bot](https://github.com/vllm-bot)
- **Merged time:** 2026-05-06 01:11:38
- **Type:** `SUBSTANTIAL`
- **[Changed Files](https://github.com/vllm-project/vllm/pull/40726/files) (4):**
  - `tests/compile/test_codegen.py`
  - `vllm/compilation/backends.py`
  - `vllm/compilation/caching.py`
  - `vllm/compilation/codegen.py`

### Summary

**What changed and why**  
Fixed codegen for unqualified names (e.g., torch.device, torch.dtype) by lifting non-primitive constants into a __vllm_consts__ list instead of relying on repr(), which caused NameError at runtime. The change modifies _node_ref to check for primitive types and store others in a shared consts list, updates generate_execution_code and compile_execution_fn to return/accept consts, and plumbs consts through backends.py and caching.py. Added 13 unit tests in test_codegen.py.

**Technical impact**  
Generated execution code now references non-primitive constants by index (e.g., __vllm_consts__[0]) instead of inline repr(), making it eval-able in a namespace with only torch and operator. The consts list is deduplicated by identity and shared across inlined submodules. Legacy cached code without consts is handled by defaulting to None. The change affects all codegen paths and requires updating the serialization format for cached artifacts.

**Potential risks**  
Risk of breaking backward compatibility with cached compilation artifacts that lack consts; the code handles this by defaulting consts to None. The identity-based dedup assumes objects are live during codegen and picklable for caching; if an object's id is reused or it is not picklable, it could cause errors. The change also increases memory usage by keeping consts alive for the lifetime of the compiled function.

**Key insights**  
The root cause was that repr() of objects like torch.device produces unqualified names (e.g., device(type='cpu')) which are not valid in the exec namespace. The fix mirrors the existing __vllm_submods__ pattern for submodules, using a list of constants referenced by index. The 13 tests cover primitive inlining, device/dtype/dict/tuple/list/positional lifting, dedup, empty consts, and end-to-end scenarios including legacy backward compatibility.

### Summary

**What changed and why**  
This PR enables W8A8 INT8 quantization on AVX2 CPUs by linking the _C_AVX2 shared library against dnnl_ext, adding AVX2-compatible vector type implementations (e.g., INT8Vec16, FP32Vec16 partial operations), and fixing a loop increment bug in dnnl_kernels.cpp. Previously, these features were gated behind __AVX512F__, causing missing-symbol errors on AVX2-only hosts like Xeon-6 E-cores. The change also compiles dnnl_ext with -mavx2 flags and removes the AVX512-only restriction.

**Technical impact**  
AVX2 hosts can now run W8A8 INT8 quantized models, achieving ~172.7 toks/s vs 72.5 toks/s for bf16 on the tested platform. The dnnl_ext library is now compiled with AVX2 flags, but oneDNN's runtime ISA detection ensures JIT-compiled kernels remain optimal. The change adds fallback code for missing AVX512 intrinsics (e.g., masked stores) and introduces new vector operations (clamp, abs, min, max, reduce_min, negation) for FP32Vec16, as well as an INT8Vec16 type for AVX2.

**Potential risks**  
Compiling dnnl_ext with -mavx2 may produce suboptimal code on AVX512 systems, but oneDNN's JIT compilation should mitigate this. The fallback implementations for masked stores and partial min/max use scalar loops, which could be slower on large tail sizes. The loop increment fix in dnnl_kernels.cpp corrects a bug that previously caused incorrect behavior on all architectures. The new AVX2 code paths are untested on non-AVX2 x86 CPUs; however, the build system only enables them when AVX2 is available.

**Key insights**  
The PR demonstrates a pragmatic approach to extending AVX2 support by reusing existing dnnl_ext and adding only the necessary vector type abstractions. The performance gain (2.4x over bf16) highlights the importance of INT8 quantization on AVX2, where bf16/fp16 run at fp32 rates. The fallback patterns for missing AVX512 intrinsics are a common technique for maintaining compatibility across x86 ISA levels.
