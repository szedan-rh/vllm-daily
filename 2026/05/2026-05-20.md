  - `vllm/transformers_utils/config.py`

### Summary

**What changed and why**  
Added a fallback in get_safetensors_params_metadata to read safetensors metadata from the local HuggingFace cache when the Hub fetch fails (e.g., 429 rate limit). Previously, a failed Hub call returned empty metadata, causing _resolve_auto_dtype to incorrectly default to bf16 on bf16-capable platforms even when the cached checkpoint was fp16. This led to flaky CI test failures for bigscience/bloom-560m due to logit tie-breaking differences between bf16 and fp16.

**Technical impact**  
Improves robustness of dtype auto-detection by using locally cached safetensors metadata when the Hub is unreachable. Ensures correct dtype (fp16) is used when the checkpoint is cached locally, preserving the existing fallback to bf16 only when cache is also missing. Reduces flakiness in CI tests that depend on Hub availability.

**Potential risks**  
The new fallback calls huggingface_hub.snapshot_download with local_files_only=True, which may raise LocalEntryNotFoundError if the cache is incomplete or missing. This is caught and logged, returning empty metadata, which preserves the previous behavior. However, if the local cache has a different revision than expected, the metadata might be stale. The change is limited to the metadata lookup path and does not affect weight loading.

**Key insights**  
The root cause was a silent fallback to bf16 when Hub metadata was unavailable, even though the model weights were cached locally in fp16. The fix leverages the existing local cache to retrieve metadata, aligning dtype resolution with the actual cached weights. This is a targeted fix that only affects the metadata retrieval path and does not change the overall architecture.
