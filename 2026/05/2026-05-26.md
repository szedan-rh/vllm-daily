### Summary

**What changed and why**  
Implements the MooncakeStoreConnector.reset_cache() method to cascade prefix cache resets to the external Mooncake master via a new ZMQ admin protocol. Previously reset_cache was a no-op, leaving stale KV blocks from old weights in the external store, which caused silent correctness bugs in RL post-training workflows. The change adds a named-tag wire protocol (protocol.py), a reset_store() method on MooncakeStoreScheduler, a reset() method on LookupKeyClient, and a RESET_MSG dispatch branch in LookupKeyServer that drains the send queue and calls store.remove_all(force=True). Also fixes reset_connector_cache() to return True when no connector is configured (previously returned False, causing spurious failures).

**Technical impact**  
Adds 8 new files/modifications (+517/-14 lines). Introduces a new protocol module for wire-format constants, extends the existing ZMQ admin channel with a typed tag system, and adds a full reset cascade from Scheduler through connector to the external store. The change is backward-compatible for existing lookups (LOOKUP_MSG tag added to requests). The reset RPC is soft-fail (returns False on error) to avoid crashing training loops. The no-connector fix changes behavior of reset_connector_cache() from returning False to True, which may affect callers that relied on the old return value.

**Potential risks**  
The reset ordering assumption (no in-flight lookups/transfers) is documented but not enforced; misuse could lead to stale KV after reset. The protocol change adds a tag frame to all lookup requests, which could break if there are other clients or versions of LookupKeyServer that don't expect the tag. The drain-then-remove_all sequence in LookupKeyServer could deadlock if the send thread is blocked. The no-connector return value change could mask configuration errors where a connector was expected but not set up. Integration tests rely on CI; local testing was limited due to environment issues.

**Key insights**  
The design reuses the existing ZMQ admin channel to avoid new sockets, and uses a named-tag protocol for extensibility. The drain-before-remove_all pattern is critical to prevent stale puts from repopulating the store after reset. The soft-fail approach (returning False instead of raising) aligns with the existing error-handling pattern in the codebase. The fix for no-connector reset returning True is a subtle but important correctness improvement for callers that unconditionally request connector resets.

The code comment already notes that subclasses with `apply_input_quant=False` must not use `As`, so the risk is minimal.

This highlights how even small overheads from tensor initialization can impact performance in high-throughput inference. The timeout and error handling in gather_actual_addresses should prevent hangs. The risk of regression is low given the extensive testing.

**Key insights**  
The root cause is a classic TOCTOU race between port probing and binding. The fix leverages ZMQ's ability to bind to port 0 and retrieve the assigned port atomically.
