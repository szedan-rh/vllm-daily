**Technical impact**  
Reduces latency in pooling forward pass by removing unnecessary device synchronizations. The changes ensure that data-dependent operations (e.g., computing segment_ids, splitting hidden states, masking with step_tag_id) are performed on CPU without blocking GPU execution, improving throughput.

**Key insights**  
The PR targets a common performance pitfall: implicit GPU<->CPU syncs from operations like .tolist(), .item(), or boolean indexing on GPU tensors. By keeping data on CPU when possible and using non_blocking transfers, the code avoids stalling the GPU pipeline. The approach is systematic and leverages existing CPU-side metadata.

## 9. [[Model Runner V2] Rebuild attn metadata between draft decode steps](https://github.com/vllm-project/vllm/pull/41162)


### Base Information

- **PR Number:** #41162
- **Author:** [TheEpicDolphin](https://github.com/TheEpicDolphin)
- **Merged By:** [WoosukKwon](https://github.com/WoosukKwon)
- **Merged time:** 2026-05-04 17:44:56
- **Type:** `SUBSTANTIAL`
- **[Changed Files](https://github.com/vllm-project/vllm/pull/41162/files) (3):**
  - `vllm/v1/worker/gpu/sample/gumbel.py`
  - `vllm/v1/worker/gpu/spec_decode/eagle/speculator.py`
  - `vllm/v1/worker/gpu/spec_decode/probabilistic_rejection_sampler_utils.py`

### Summary

**What changed and why**  
This PR fixes a crash in speculative decoding (MTP > 2) by rebuilding attention metadata between draft decode steps. Previously, attention metadata was not rebuilt during multi-step draft generation, causing position-dependent state to become stale and leading to invalid memory access. The change restructures the decode loop to rebuild metadata each step, and modifies the gumbel sampling kernel to accept a column tensor for writing logits to the correct step column.

**Technical impact**  
The multi-step decode loop is now broken into individual steps with attention metadata rebuilt each iteration, losing the ability to capture the entire loop as a single cudagraph. Instead, only a single-step draft generation is captured and replayed. A scalar tensor `current_draft_step` is introduced to track the step index, and the gumbel sampling kernel is extended with a `processed_logits_col_ptr` parameter to support writing to a specific column of the logits output tensor. Performance benchmarks show minimal regression (typically <2% throughput change) across multiple models and concurrency levels.

**Potential risks**  
The change may introduce a performance regression for models with many speculative steps due to the loss of full-loop cudagraph capture. The new `processed_logits_col_ptr` parameter in the gumbel kernel adds complexity and could affect other callers (e.g., probabilistic rejection sampler) which now pass additional dummy arguments. There is a risk of incorrect column indexing if the tensor is not properly synchronized or if the step tensor is modified asynchronously.

**Key insights**  
The root cause was that attention metadata builders (e.g., for DSV4) maintain position-dependent state that must be updated when positions advance, but the draft decode loop was only updating sequence lengths and positions, not rebuilding metadata. The fix ensures correctness by rebuilding metadata each step, at the cost of cudagraph efficiency. Future work could add a cudagraph-safe in-place update method to the attention metadata builder API to recover performance.
